# 物理存储介质基础
- 计算机的存储结构：参见[[L4.1 存储器体系结构#存储器结构抽象|计算机组成原理——存储器结构抽象]]与[[L4.2 缓存友好的代码#缓存组织和操作|计算机组成原理——缓存组织和操作]]
- 存储介质特性与分类：
	- 存储评价指标：
		- 访问速度：数据读取/写入的耗时
		- 单位成本：每字节存储的成本
		- 可靠性：抗断电，抗系统崩溃和物理故障的能力
	- 介质分类：
		- 易失性存储（volatile storage）：如RAM，断电后数据丢失，包括Cache和主存
		- 非易失性存储（non-volatile storage）：如磁盘，断电后数据仍然存在，包括HDD和SSD
	- 各存储介质特性：
	- |介质类型|核心特性|访问速度|适用场景|
		|---|---|---|---|
		|**Cache**|最快、成本最高、硬件管理|纳秒级|CPU 高频数据缓存（L1/L2/L3）|
		|**Main Memory**|易失、容量数百 GB 级|几十纳秒|临时数据处理，无法存完整数据库|
		|**Flash（闪存）**|非易失、读快写慢、有限擦除周期（10K-1M 次）|读≈主存，写 = 微秒级|嵌入式设备、SSD、U 盘|
		|**Magnetic Disk（硬盘）**|磁存储、直接访问、容量极大|毫秒级（寻道 + 旋转延迟）|数据库长期存储（完整数据库），单盘最大约 44TB|
		|**Optical Disk（光盘）**|激光读取、非易失|慢于硬盘|数据分发、离线存储（CD/DVD/ 蓝光）|
		|**Magnetic Tape（磁带）**|顺序访问、极低单位成本|最慢|数据备份、海量归档（Oracle SL8500 库可达 1.2EB）|
# 磁盘
- 物理结构：参见[[L4.1 存储器体系结构#^e6f3f2|计算机组成原理——非易失存储器的类型]]
- 访问时间：参见[[L4.1 存储器体系结构#^957f04|计算机组成原理——磁盘访问时间]]
	- *注1：磁盘内道的速率比外道低，因为内道的线速度较慢*
	- *注2：接口传输率也会影响访问时间，尤其是对于大数据量的读写操作*
- 磁盘可靠性：平均失效间隔（MTTF）
	- 磁盘连续无故障运行的平均时间
		- 新盘的理论MTTF在50万-120万小时，对应3-5年的使用寿命
	- 故障规律：在使用早期和硬件老化的后期故障率上升
- 磁盘调度优化：
	- 盘块与盘臂调度优化：
		- 盘块（Block）：
			- 单个磁道上连续的扇区组成的**数据传输的基本单位**
			- 大小：一般在4k-16kB之间
				- 块过小：增加I/O次数，降低效率
				- 块过大：产生内部碎片，浪费存储空间
		- 盘臂调度优化：减少磁头的移动，降低寻道时间
			- **电梯算法**：磁头在磁道之间来回移动，优先服务当前移动方向上的请求，直到没有更多请求，然后反向移动
	- 文件碎片与优化：
		- 文件碎片：文件随操作，存储块分布在磁盘不同位置，增加寻道时间
		- 优化：
			- 将关联数据存储在相近的物理位置
			- 定期进行磁盘碎片整理，重新排列文件以减少碎片
# 存储访问
- 为高效访问存储，需要缓存机制
	- 块（Block）：数据传输的基本单位，通常为4KB-16KB
	- 缓存（Buffer）：内存中的临时存储区域，用于存储从磁盘读取的数据块
	- 缓存管理器（Buffer Manager）：负责管理缓存，处理数据块的加载、替换和写回
- 缓存管理器的工作原理：
	- 程序访问磁盘块时，会先向缓存管理器发起请求：
		- 命中（Hit）：请求的数据块已在缓存中，直接返回数据，无需磁盘I/O
		- 缺失（Miss）：请求的数据块不在缓存中，需要从磁盘读取数据块到缓存中，然后返回数据
			1. 分配缓存空间：如果缓存有空余，直接分配；如果缓存已满，需要选择一个块进行替换
				- 被替换的块，只有被修改过时（脏位为1），才需要写回磁盘，称为**延迟写回**策略
			2. 读入数据块：从磁盘读取数据块到缓存中，更新缓存状态
			3. 返回地址：返回新加载的数据块在缓存中的地址给程序
    - 缓存替换策略：
	    - LRU（Least Recently Used）：优先替换最近最少使用的块
		    - 优点：适用于访问模式具有时间局部性的情况
		    - 局限：当嵌套循环访问时，可能会频繁替换块，导致性能下降
		- Toss-Immediate（立即销毁）：每次访问后立即销毁块
            - 适合嵌套循环中的外层关系块，避免频繁替换块
        - MRU（Most Recently Used）：优先替换最近最常使用的块
            - 适合嵌套循环中的内层关系块，被反复扫描的块此时
        - 现代缓存管理：
	        - 多种替换策略结合使用
	        - 基于预测的策略：利用访问统计信息，将高频块常驻于缓存中
    - 并发场景安全访问：PIN/UNPIN操作
	    - PIN：将块固定在缓存中，防止被替换
        - UNPIN：解除块的固定状态，使其可以被替换
# 文件中的记录组织
1. 定长记录方式（Fixed-length Records）：
    - 每条记录占用固定字节数，便于计算和访问
	    - 例：如果每条记录占用100字节，那么第n条记录的起始位置可以通过公式`100 * (n - 1)`直接计算得到
    - 优点：寻址简单，访问效率高
	    - 注：由于定长可能导致一条记录位于两个磁盘块内，增加I/O次数，因此实际实现中，**禁止记录的跨块存储**】
	- 删除操作：删除记录时，需要处理空闲空间
		1. 移动后续记录整理：将删除记录后的记录向前移动，保持记录连续存储
			- 优点：保持记录连续，便于访问
			- 缺点：删除操作效率较低，尤其是当记录数量较多时，移动大量记录会导致性能下降
        2. 末尾回填：将最后一条记录移动到被删除记录的位置，保持记录连续存储
            - 优点：删除操作效率较高，仅需移动一条记录
            - 缺点：可能导致记录顺序改变，影响顺序敏感场景的性能
        3. 自由链表（Free List）：维护一个链表记录被删除记录的位置，文件头记录第一个空闲记录的位置，每个空闲记录中存储下一个空闲记录的位置；当新记录插入时，优先使用链表中的空闲位置
            - 优点：删除操作效率高，无需移动记录
            - 缺点：访问时需要额外检查链表，增加访问复杂度
2. 变长记录方式（Variable-length Records）：当本身有变长字段时，或者为了节省存储空间时，采用变长记录方式
	- 标准存储结构：以instructor表为例
		- 属性按序存储，定长在前，变长元数据居中，实际变长数据在后
		- 变长字段用`(offset, length)`表示，记录变长字段的相对偏移和长度
		- 空值使用`null`位图表示，用一段二进制数据中的每一位表示一个字段是否为null
		- 例：![[Pasted image 20260417144608.png]]
	- 实现方式：
		1. **槽页结构**（Slotted Page Structure）： ^e7d62a
			- 示意图：![[Pasted image 20260417144707.png]]
			- 将磁盘块分为三部分：块头，空闲空间，记录区（从块尾向前）
				- 块头中存储记录条目数，空闲空间的结束地址，每个记录的位置和大小
			- 优势：
				- 记录可移动且无碎片：删除中间记录，可以前移后续记录合并空余空间
				- 指针解耦：记录位置通过块头中的条目指针访问，记录移动不影响访问
		2. 用定长单元存储变长记录：
            - 预留固定长度的空间，用指针连接，存储变长数据
            - 适用场景：不支持记录移动的系统，实现简单，但利用率和性能不如槽页结构
        3. 最大长度记录（伪变长记录）：
            - 将变长字段的最大长度作为定长字段存储
            - 适用于变长字段长度变化不大的场景，简化实现，但非常浪费存储空间
# 文件组织
- 文件组织方式：
	- 堆文件（Heap File）：
        - 结构：记录无序存储，插入时直接追加到文件末尾
        - 优点：插入效率高，无需维护索引
        - 缺点：查询效率低，尤其是没有索引的情况下，需要扫描整个文件
    - [[L3 存储与文件结构#^ad9450|顺序文件]]（Sequential File）：
        - 结构：记录按搜索键的值排序存储，插入时需要保持排序
        - 优点：顺序处理快，查询效率高，尤其是范围查询
        - 缺点：插入效率低，需要移动大量记录以保持排序
    - 哈希文件（Hash File）：
        - 结构：根据搜索键的值通过哈希函数计算存储位置，记录分布在不同的桶中
        - 优点：查询效率高，尤其是等值查询
        - 缺点：范围查询效率低，哈希冲突可能导致性能下降、
- 文件模式：
	- 小文件模式（Single-file per relation）：
		- 每个关系对应一个文件，文件中存储该关系的所有记录
		- 优点：单表读取效率高，适合单表查询
		- 缺点：多表查询效率低，尤其是连接操作，需要频繁访问多个文件，多次I/O导致效率低下
    - [[L3 存储与文件结构#^27a2fc|大文件模式]]（Multi-table Clustering）：
        - 将多个相关的关系存储在同一个文件中，记录按照某个公共属性（如主键）排序存储，也叫**多表聚集文件组织**
- 顺序文件结构与操作： ^ad9450
	- 结构特点：按照某种搜索键的升序或降序顺序存储
		- 例：![[Pasted image 20260417150713.png]]
	- 删除操作：使用指针链处理
		- 不直接移动记录，而是将被删除记录的前一个记录的指针指向被删除记录的下一个记录，形成链表结构
	- 插入操作：
		- 对应检索位置有空闲时，直接插入到对应位置
		- 对应位置没有空闲时，插入到**溢出块**中，再通过指针链接到主块
			- 例：![[Pasted image 20260417150925.png]]
	- 维护：定期重组文件，将溢出块中的记录重新插入到主块中，保持文件的有序性和查询效率
- 多表聚集结构： ^27a2fc
	- 结构特点：将多个相关的关系存储在同一个文件中，按照关联的字段聚集存储
		- 例：![[Pasted image 20260417151051.png]]
		- 多表查询效率快，但单表查询时跟随大量冗余记录，浪费I/O
			- 优化：单表指针链。给每个关系维护一个指针链，指向同一主键值的记录，单表查询时只需访问对应关系的记录，避免冗余记录的访问

# 分布式文件系统
- 大数据时代，大量的数据难以存储在同一台极其上，因此，分布式文件系统（Distributed File Systems，DFS）成为现代大数据的数据库系统的基础
	- 特点：对外提供单一文件系统视图，可扩展性强，解决大量硬件的不可靠问题
	- 例：
		- Google File System（GFS）：Google开发的分布式文件系统，设计用于大规模数据处理，提供高吞吐量和容错能力
		- Hadoop File System（HDFS）：Apache Hadoop项目的分布式文件系统，基于GFS设计，广泛应用于大数据处理领域
	- GFS的设计原则：
		- 采用一主多从架构，主服务器负责管理文件系统的元数据，从服务器负责存储实际数据块
		- 控制流与数据流分离：客户端直接与数据服务器进行数据传输，减少主服务器的负载
		- 分片与副本机制：将文件分成固定大小的块（如64MB），每块存储多个副本（如3份），提高数据可靠性和访问效率
	- Hadoop系统：开源维护，基于GFS设计，提供了分布式存储和计算框架，支持大规模数据处理任务，如MapReduce、Spark等
