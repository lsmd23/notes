# 事务的基本概念
- 事务（Transaction）：访问并可能更新数据项的执行单元，也是保证数据一致性和可靠性的基本单元
	- 核心问题：
		- 故障恢复：当系统发生故障时，如何保证数据的稳定
		- 并发控制：当多个事务同时访问数据时，如何保证事务处理的正确性
	- 例：银行转账事务——从账户A向账户B转账\$50
		- `read(A)`：读取账户A的余额
		- `A = A - 50`：从账户A扣除\$50
		- `write(A)`：将更新后的余额写回账户A
		- `read(B)`：读取账户B的余额
		- `B = B + 50`：向账户B增加\$50
		- `write(B)`：将更新后的余额写回账户B
- 事务的特性（ACID）：
	1. 原子性（Atomicity）：事务中的所有操作要么全部执行成功，要么全部不执行，不能只执行一部分
		- 假设转账事务在执行过程中被中断，如果只执行了`read(A)`和`A = A - 50`，但没有执行`write(A)`，则账户A的余额会不正确
	2. 一致性（Consistency）：事务执行前后，数据库满足同样的完整性约束
		- 显式约束：数据库本来要求的约束，如字段的数据类型、唯一性等 
        - 隐式约束：应用程序中定义的约束，如转账事务中要求账户余额不能为负数，转账前后总额不变等
    3. 隔离性（Isolation）：事务的执行不受其他事务的干扰，每个事务都像是在独立运行一样
	    - 要求：多个事务并发执行时，应当和串行执行的结果相同
        - 例如，如果转账事务和查询事务同时进行，查询事务不应该看到转账事务未完成的中间状态（如账户A已经扣款但账户B还未加款）
    4. 持久性（Durability）：一旦事务提交，其结果就是永久性的，即使系统发生故障也不会丢失
# 事务状态
- 存储结构：参见[[L3 存储与文件结构|存储结构]]
	- 事务执行的中间数据存储于易失存储中
	- 事务执行的结果（也即数据库的最终状态）存储于非易失存储中
	- **可靠存储**：通过多副本实现的稳定数据，不会因为系统故障而丢失，用于存储事务的日志，从而实现事务的持久性
- 事务模型：简化事务模型，忽略SQL操作中复杂的`INSERT/DELETE`等操作，只关注读写数据项的操作
	- `read(X)`：读取数据项X的值并存入事务的局部变量中
	- `write(X)`：将事务的局部变量中的值写回数据项X，并更新数据库的状态
- 事务状态：
	- 状态的分类：
		- 活跃状态（Active）：事务正在执行中，尚未完成
	    - 部分提交状态（Partially Committed）：事务已经完成了所有操作，但修改尚未写回数据库
	    - 提交状态（Committed）：事务已经完成并且修改已经写回数据库，事务成功结束
	    - 失败状态（Failed）：事务在执行过程中发生错误，无法继续执行，需要回滚
	    - 终止状态（Aborted）：事务已经回滚，恢复到执行前的状态
    - 状态转移图：![[Pasted image 20260508173741.png]]
	    - 正常执行：Active -> Partially Committed -> Committed
		    1. 事务启动后进入Active状态，执行所有读写操作
		    2. 当事务完成所有操作后进入Partially Committed状态，此时系统尝试将修改写回数据库
		    3. 如果写回成功，事务进入Committed状态，成功结束
		- 异常回滚：Active -> Failed -> Aborted
            1. 如果事务在执行过程中发生错误（如违反约束、死锁等），进入Failed状态
            2. 系统回滚事务，将数据库恢复到执行前的状态，进入Aborted状态，事务结束
	        3. 若为临时故障，可以重启事务；若逻辑发生错误，则直接终止事务
	- 已提交事务的补偿：
		- 一旦事务提交，就无法回滚以撤销其影响
		- 此时，只能通过**补偿事务**（Compensating Transaction）来执行相反的操作以达到撤销效果
		- 这部分操作由用户负责，不由数据库系统管理
# 并发执行
- 定义：多个事务在系统上宏观同时运行，微观上交替使用CPU/IO资源（Concurrent Execution）
	- 优点：提升系统资源利用率，缩短事务的平均响应时间
	- 区分：并发执行不等同于并行执行（Parallel Execution），并行执行是指多个事务在不同的处理器上同时运行，而并发执行可能在单处理器上交替运行
