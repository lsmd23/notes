# 事务的基本概念
- 事务（Transaction）：访问并可能更新数据项的执行单元，也是保证数据一致性和可靠性的基本单元
	- 核心问题：
		- 故障恢复：当系统发生故障时，如何保证数据的稳定
		- 并发控制：当多个事务同时访问数据时，如何保证事务处理的正确性
	- 例：银行转账事务——从账户A向账户B转账\$50
		- `read(A)`：读取账户A的余额
		- `A = A - 50`：从账户A扣除\$50
		- `write(A)`：将更新后的余额写回账户A
		- `read(B)`：读取账户B的余额
		- `B = B + 50`：向账户B增加\$50
		- `write(B)`：将更新后的余额写回账户B
- 事务的特性（ACID）：
	1. 原子性（Atomicity）：事务中的所有操作要么全部执行成功，要么全部不执行，不能只执行一部分
		- 假设转账事务在执行过程中被中断，如果只执行了`read(A)`和`A = A - 50`，但没有执行`write(A)`，则账户A的余额会不正确
	2. 一致性（Consistency）：事务执行前后，数据库满足同样的完整性约束
		- 显式约束：数据库本来要求的约束，如字段的数据类型、唯一性等 
		- 隐式约束：应用程序中定义的约束，如转账事务中要求账户余额不能为负数，转账前后总额不变等
	3. 隔离性（Isolation）：事务的执行不受其他事务的干扰，每个事务都像是在独立运行一样
		- 要求：多个事务并发执行时，应当和串行执行的结果相同
		- 例如，如果转账事务和查询事务同时进行，查询事务不应该看到转账事务未完成的中间状态（如账户A已经扣款但账户B还未加款）
    4. 持久性（Durability）：一旦事务提交，其结果就是永久性的，即使系统发生故障也不会丢失
# 事务状态
- 存储结构：参见[[L3 存储与文件结构|存储结构]]
	- 事务执行的中间数据存储于易失存储中
	- 事务执行的结果（也即数据库的最终状态）存储于非易失存储中
	- **可靠存储**：通过多副本实现的稳定数据，不会因为系统故障而丢失，用于存储事务的日志，从而实现事务的持久性
- 事务模型：简化事务模型，忽略SQL操作中复杂的`INSERT/DELETE`等操作，只关注读写数据项的操作
	- `read(X)`：读取数据项X的值并存入事务的局部变量中
	- `write(X)`：将事务的局部变量中的值写回数据项X，并更新数据库的状态
- 事务状态：
	- 状态的分类：
		- 活跃状态（Active）：事务正在执行中，尚未完成
		- 部分提交状态（Partially Committed）：事务已经完成了所有操作，但修改尚未写回数据库
		- 提交状态（Committed）：事务已经完成并且修改已经写回数据库，事务成功结束
		- 失败状态（Failed）：事务在执行过程中发生错误，无法继续执行，需要回滚
		- 终止状态（Aborted）：事务已经回滚，恢复到执行前的状态
	- 状态转移图：![[Pasted image 20260508173741.png]]
	    - 正常执行：Active -> Partially Committed -> Committed
		    1. 事务启动后进入Active状态，执行所有读写操作
		    2. 当事务完成所有操作后进入Partially Committed状态，此时系统尝试将修改写回数据库
		    3. 如果写回成功，事务进入Committed状态，成功结束
		- 异常回滚：Active -> Failed -> Aborted
            1. 如果事务在执行过程中发生错误（如违反约束、死锁等），进入Failed状态
            2. 系统回滚事务，将数据库恢复到执行前的状态，进入Aborted状态，事务结束
	        3. 若为临时故障，可以重启事务；若逻辑发生错误，则直接终止事务
	- 已提交事务的补偿：
		- 一旦事务提交，就无法回滚以撤销其影响
		- 此时，只能通过**补偿事务**（Compensating Transaction）来执行相反的操作以达到撤销效果
		- 这部分操作由用户负责，不由数据库系统管理
# 并发执行
- 定义：多个事务在系统上宏观同时运行，微观上交替使用CPU/IO资源（Concurrent Execution）
	- 优点：提升系统资源利用率，缩短事务的平均响应时间
	- 区分：并发执行不等同于并行执行（Parallel Execution），并行执行是指多个事务在不同的处理器上同时运行，而并发执行可能在单处理器上交替运行
	- 并发控制：并发需要保证事务的ACID性质，特别是隔离性，避免事务之间的干扰导致数据不一致
- 调度（Schedule）：
	- 定义：一个指令序列，指定并发事务中所有指令执行的时间顺序
	- 规则：
		- 完整性：一个调度必须包含所有参与事务的指令
		- 保序性：一个调度必须保持每个事务内部指令的相对顺序不变、
	- 调度示例：
		- 事务T1：从账户A转账\$50到账户B：`read(A)`, `A = A - 50`, `write(A)`, `read(B)`, `B = B + 50`, `write(B)`, `commit`
		- 事务T2：从账户A转账余额的10%到账户B：`read(A)`, `temp = A * 0.1`, `A = A - temp`, `write(A)`, `read(B)`, `B = B + temp`, `write(B)`, `commit`
		- 串行调度两事务时，`T1 → T2`的调度顺序和`T2 → T1`的调度顺序可能导致不同的结果，但都是正确的调度（二者总和不变）
		- 并发调度时，需要注意调度指令的顺序有可能导致数据不一致，如`T1`和`T2`的指令交错执行可能导致账户A的余额被错误地更新
			- 调度1：![[Pasted image 20260508195235.png]]
			- 调度2：![[Pasted image 20260508195250.png]]
			- 前者是等价调度，后者则不是等价调度
# 可串行化
- 串行调度与可串行化：
	- 基本假设：每个事务单独执行时，都能保证数据库的一致性，因此，串行调度一定是正确的调度
		- 可串行化：一个并发调度，如果和某个串行调度等价（即两者的结果相同），则称该并发调度是可串行化的（Serializable）
			- 冲突可串行化（Conflict Serializable）：参见[[L7 事务#^c92e5c|后文]]
			- 视图可串行化（View Serializable）：参见[[L7 事务#^97b64e|后文]]
- 冲突可串行化： ^c92e5c
	- 指令冲突：两个不同事务的指令$I_i$和$I_j$，若它们访问同一个数据项$Q$，且至少有一个指令是`write(Q)`，则称$I_i$和$I_j$存在冲突
		- 只有两个读指令之间没有冲突，其他组合（读-写、写-读、写-写）都存在冲突
		- **冲突指令的顺序交换会改变调度的结果**，而非冲突指令的顺序交换不会改变调度的结果
	- **冲突等价**：如果一个调度$S$可以通过交换非冲突指令得到另一个调度$S'$，则称$S$和$S'$是冲突等价的
	- **冲突可串行化**：如果调度$S$和某个串行调度$S'$是冲突等价的，则称$S$是冲突可串行化的
		- 例：![[Pasted image 20260508200020.png]]这两个调度是冲突等价的，也即调度3是可冲突串行化的![[Pasted image 20260508200121.png]]这个调度则是不可冲突串行化的
	- **先序图**（Precedence Graph）：一个有向图，用于判断调度是否冲突可串行化
        - 顶点：每个事务对应一个顶点
        - 边：如果事务$T_i$中的指令$I_i$和事务$T_j$中的指令$I_j$存在冲突，并且$I_i$在调度中先于$I_j$执行，则从顶点$T_i$到顶点$T_j$画一条有向边
        - 判断：**一个调度是冲突可串行化的，当且仅当其先序图是无环的**，此时可以通过拓扑排序得到一个串行调度与之冲突等价
- 视图可串行化： ^97b64e
    - 视图等价：如果两个调度$S$和$S'$满足以下条件，则称它们是视图等价的：
        1. **读初始值一致**：若事务$T_i$在$S$中读取了数据项$Q$的初始值，则在$S'$中也必须读取$Q$的初始值
        2. **读来源一致**：若事务$T_i$在$S$中读取了数据项$Q$的值，并且该值是由事务$T_j$写入的，则在$S'$中也必须读取由$T_j$写入的$Q$的值
        3. **最终写一致**：如果事务$T_i$在$S$中最后写入了数据项$Q$，则在$S'$中也必须由$T_i$最后写入$Q$
    - 视图可串行化：如果调度$S$和某个串行调度$S'$是视图等价的，则称$S$是视图可串行化的
        - 视图可串行化是冲突可串行化的超集，即所有冲突可串行化的调度都是视图可串行化的，但并非所有视图可串行化的调度都是冲突可串行化的
        - 例：![[Pasted image 20260508200628.png]]调度7是视图可串行化的，但不是冲突可串行化的
    - 判定：视图可串行化的判定是一个[[L9 NP完全性#NP-完全性|NP完全问题]]，没有高效的算法可以判断一个调度是否是视图可串行化的
- 其它的等价调度：
	- 例：![[Pasted image 20260508200852.png]]这一调度和串行的调度结果一致，但既不属于冲突可串行化，也不属于视图可串行化的调度
	- 带有算术运算的调度，不能被简单的简化为读写指令的调度，因此无法使用前面定义的冲突或视图等价来判断其是否可串行化
# 可恢复性
- 事务的结束：
	- 提交（Commit）：事务成功完成，所有修改都写回数据库，之后执行提交指令，事务结束
	- 中止（Abort）：事务执行过程中发生错误，无法继续执行，需要**回滚**（Roll Back）事务，将数据库恢复到执行前的状态，之后执行中止指令，事务结束
		- 回滚事务涉及事务的可恢复性问题
- 事务的调度与恢复：
    - 可恢复调度（Recoverable Schedule）：如果一个事务$T_j$读取了另一个事务$T_i$之前写入的数据项，则事务$T_i$的提交必须在事务$T_j$的提交之前，才是可恢复的调度
	    - 例：![[Pasted image 20260524160508.png]]该调度中$T_9$读入了$T_8$写入的A，但$T_9$在$T_8$提交之前就提交了，因此该调度不是可恢复的
	- 级联回滚（Cascading Rollback）：如果一个事务$T_j$读取了另一个事务$T_i$之前写入的数据项，并且事务$T_i$在执行过程中发生错误需要回滚，那么事务$T_j$也必须回滚，这种情况称为级联回滚
        - 例：![[Pasted image 20260524160910.png]]
	        - 这是一个可恢复的调度，因为三个事务都尚未提交
	        - $T_{10}$最后进入中止态，其写入A，会导致后续读取A的事务$T_{11}$需要回滚，从而引发级联回滚
	    - 级联回滚还会导致大量已经完成的工作被撤销，严重影响系统的效率
	- 无级联回滚的调度（Cascade-Less Schedule）：对每一对$T_i$和$T_j$，如果$T_j$读取了$T_i$之前写入的数据项，则事务$T_i$必须在事务$T_j$提交之前提交，这样的调度称为无级联回滚的调度
		- **每个无级联回滚的调度都是一个可恢复的调度**
# 隔离级别
- 并发一致性问题：
    - 脏读（Dirty Read）：一个事务**读取了另一个事务尚未提交的修改**，如果后者回滚，则前者读取的数据就是无效的
	    - 例：![[Pasted image 20260524212701.png]]脏读导致的回滚
    - 不可重复读（Non-Repeatable Read）：一个事务在两次读取同一数据项时，**另一个事务修改了该数据项并提交**，导致前者两次读取的结果不同
	    - 例：![[Pasted image 20260524212924.png]]不可重复读导致的错误
    - 幻读（Phantom Read）：一个事务在**两次执行相同的查询时，另一个事务插入了满足查询条件的新记录**，导致前者两次查询的结果不同异常
	    - 例：![[Pasted image 20260524212956.png]]幻读导致的错误
- SQL-92定义的隔离级别：
	- 为解决数据一致性和并发性能之间权衡的问题，SQL-92标准定义了四种隔离级别，分别允许不同程度的并发异常：
        1. 读未提交（Read Uncommitted）：允许脏读、不可重复读和幻读
        2. 读已提交（Read Committed）：禁止脏读，但允许不可重复读和幻读
        3. 可重复读（Repeatable Read）：禁止脏读和不可重复读，但允许幻读
        4. 可串行化（Serializable）：禁止所有并发异常，提供最高的隔离级别，但性能最低
    - 隔离级别的实现：
	    1. 锁（Locking）：通过加锁机制控制事务对数据项的访问，确保不同事务之间的操作不会互相干扰
            - 共享锁（Shared Lock）：允许多个事务同时读取数据项，但不允许修改，也叫读锁
            - 排他锁（Exclusive Lock）：只允许一个事务访问数据项，既不允许其他事务读取也不允许修改，也叫写锁
         2. 时间戳机制（Timestamps）：为每个事务分配一个唯一的时间戳，按照时间戳的顺序来控制事务的执行，确保事务之间的操作不会互相干扰
         3. 多版本机制（Multiple Versions）：为每个数据项维护多个版本，允许事务访问不同版本的数据项，从而避免锁的竞争，提高并发性能
# SQL中的事务
- 事务的开启：
	- 语句：`BEGIN TRANSACTION`或`START TRANSACTION`
	- 执行该语句后，后续所有DML操作都会被视为一个事务的一部分，直到执行`COMMIT`或`ROLLBACK`语句
- 事务的提交：
    - 语句：`COMMIT`
    - 执行该语句后，事务中的所有修改都会被永久保存到数据库中，事务结束
- 事务的回滚：
    - 语句：`ROLLBACK`
    - 执行该语句后，事务中的所有修改都会被撤销，数据库恢复到事务开始之前的状态，事务结束
- 自动提交模式：
	- 默认情况下，几乎所有数据库系统都启用了自动提交模式，即每个单独的SQL语句都被视为一个事务，并在执行后自动提交
	- 可以通过设置`AUTOCOMMIT`参数来启用或禁用自动提交模式
