- 上层工具发送的SQL请求，都交由查询处理器（Query Processor）来处理，查询处理器的职责是解析、优化和执行SQL请求
- 查询执行引擎（Query Execution Engine）是查询处理器的一个组件，负责执行查询计划并返回结果，同时尽可能地优化查询性能
# 概述
- 基础：[[L1 关系模型#关系代数基础|关系模型——关系代数基础]]
- 查询处理的步骤：
	1. 语法分析与翻译（Parser and Translator）：将SQL查询转换为关系代数表达式
		- Parser会首先进行语法检查，验证语法正确性
		- 之后会进行语义验证，确保查询中的表、列等元素存在且正确使用
		- 最后会将SQL翻译为扩展关系代数表达式，或者对应的查询树
		- 例：SQL语句：`SELECT name FROM instructor WHERE salary < 75000;`将被翻译成关系代数：$\Pi_\text{name}(\sigma_{\text{salary} < 75000}(\text{instructor}))$，同时生成对应的查询树
	2. 查询优化（Optimizer）：在所有查询方案中，选出代价最低的一个
		- 逻辑优化（代数优化）：对关系代数表达式做等价变换，使其更加高效
			- 例：关系代数表达式$\Pi_\text{name}(\sigma_{\text{salary} < 75000}(\text{instructor}))$和$\sigma_{\text{salary} < 75000}(\Pi_\text{name}(\text{instructor}))$是等价的，但前者通常更高效，因为它先过滤掉不满足条件的元组，再进行投影
		- 物理优化：选择具体的算法和数据结构来执行查询计划
			- 例：选择操作有全表扫描和索引扫描两种算法，优化器会根据表大小和索引清空选择更高效的算法
		- 执行计划（Evaluation Plan）：优化器生成一个带具体策略的关系代数表达式
	3. 查询执行（Evaluation Engine）：执行查询计划，调用存储管理器接口并返回结果
# 查询代价的度量
- 定义：查询代价是执行查询所需的资源消耗的度量，通常以时间、CPU使用率、内存使用率、磁盘I/O等指标来衡量
- 代价的组成：查询总代价 = **寻道时间** + **读/写块传输时间**
	- 寻道代价：$S \times t_S$，其中$S$是寻道次数，$T_s$是单次寻道的时间
	- 读代价：$b_r \times t_T$，其中$b_r$是读取的块数，$T_r$是单块传输的时间
	- 写代价：$b_w \times t_W$，其中$b_w$是写入的块数，$T_w$是单块传输的时间
		- 写代价一般远大于读代价，因为写后需要读回验证数据的正确性
- 简化查询代价：只关注块传输次数和寻道次数
	- 公式：$\text{Cost} = b\times t_T + S \times t_S$
	- $b$：块传输数
	- $S$：寻道次数
	- $t_T$：单块传输时间
	- $t_S$：单次寻道时间
- 最坏代价估计：
	- 不同缓存策略可以优化磁盘的I/O用时
	- 由于缓存策略和缓存大小都会影响实际的性能，因此在估计查询代价时，通常会采用最坏代价估计，即假设每次操作只有最小的可用存储器空间，或者每次操作都需要访问磁盘上的数据块
# 选择操作
- 索引机制：[[L4 索引#顺序索引|索引——主索引与次级索引]]
## 单条件查找选择
1. A1算法：线性扫描数据文件
	- 逐块扫描数据检验是否满足选择条件
	- A1-1算法：无索引非键属性的多结果查询
		- 只需一次寻道，读取表的所有$b_r$个块
		- 代价：$t_S + b_r \times t_T$
	- A1-2算法：无索引键属性的等值查询
		- 找到第一个满足条件的记录即停止，平均读取块数为$\frac{b_r}{2}$
		- 代价：$t_S + \frac{b_r}{2} \times t_T$
	- 特点：适用范围广，但代价较高，尤其是对于大表来说，线性扫描可能非常慢
2. A2算法：基于主索引的键属性等值查找
	- 条件：主索引建立在键属性上（如主键`ID`），数据文件按键顺序存储
	- 原理：先定位索引项，再通过指针读取
	- 代价：$(h_i+1) \times (t_S + t_T)$
		- $h_i$：主索引的高度，也即B+树的层数
		- 含义：需要$h_i$次寻道访问索引层，最后一次寻道访问数据块
3. A3算法：基于主索引的非键属性等值查找
    - 条件：主索引建立在非键属性上（如`age`），一个属性可能对应多条记录
    - 原理：先定位索引项，再顺序读取连续的磁盘块比对
    - 代价：$h_i \times (t_S + t_T) + b \times t_T + t_S$
        - $h_i$：主索引的高度
        - $b_r$：满足条件的记录所在的块数
        - 需遍历索引$h_i$次，最后一次寻道访问数据块，读取满足条件的所有$b$个块
4. A4算法：基于次级索引的等值查找
	- 目标属性上建立刺激索引，一个属性值可能对应多条记录
	- 原理：先在次级索引中定位索引项，再通过指针访问数据块，次级索引的非聚集性导致记录可能分散在不同块
	- A4-1算法：键属性等值查询
		- 代价：$(h_i+1)\times (t_S + t_T)$
		- 类似A2算法，只查询一条数据
	- A4-2算法：非键属性等值查询
        - 代价：$(h_i+n) \times (t_S + t_T)$
        - $n$为匹配的记录数，每条记录都可能对应一次寻道，代价高
5. A5/A6算法：基于索引的范围查找
	- 基于主/次级索引做范围查询（如$σ_{A≥v}​(r)$、$σ_{A≤v}​(r)$）
	- A5算法：基于主索引，且数据按序存储
		- 查询逻辑：
			- $σ_{A≥v}​(r)$：利用索引查找第一个满足条件的记录，再顺序读取后续块
			- $σ_{A≤v}​(r)$：顺序扫描，直至找到第一个不满足条件的记录
		- 代价：$h_i \times (t_S + t_T) + b \times t_T + t_S$
	- A6算法：基于次级索引，且数据不按序存储
        - 查询逻辑：
            - $σ_{A≥v}​(r)$：利用索引查找第一个满足条件的记录，再按指针读取后续块
            - $σ_{A≤v}​(r)$：顺序扫描索引叶子节点取指针，之后依次读取记录
        - 代价：$(h_i+n) \times (t_S + t_T)$
    - 注：次级索引范围扫描代价极高（每条记录都可能对应一次I/O）， 不一定比线性扫描更快
- 例：
	- 情境：
		- student 表：100000 条记录，4000 个磁盘块；ID 为主键，B + 树索引度数 100（1 个节点 = 1 块）；
		- 硬件 1：普通硬盘 tS​=4 ms,tT​=0.1 ms；
		- 硬件 2：SATA 固态 tS​=0.09 ms,tT​=0.01 ms；
		- 查询：查 ID=202301001 的学生姓名（主键等值查询）
	- 算法A1-2：线性扫描
		- 普通硬盘：$\text{Cost}=1×4+(4000/2)×0.1=4+200=204\text{ms}$
		- 固态硬盘：$\text{Cost}=1×0.09+(4000/2)×0.01=0.09+20=20.09\text{ms}$
	 - 算法 A2（B+树主索引）
		- 计算B+树高度：$h_i​=⌈log100​100000⌉=⌈5⌉=5$
		- 普通硬盘：$\text{Cost}=(5+1)×(4+0.1)=6×4.1=16.4\text{ms}$
		- 固态硬盘：$\text{Cost}=(5+1)×(0.09+0.01)=6×0.1=0.4\text{ms}$
## 复合条件查找选择
- A7/A8/A9：合取条件查找（$\sigma_{\theta_1\wedge\theta_2}(r)$）
	- 思路：优先过滤索引以减少I/O
	- A7算法：单个索引可用，其余条件没有对应的索引
        - 先使用可用索引，借助A1-A6算法过滤满足条件的记录，再对结果进行线性扫描验证其他条件
        - 代价：仅对有索引的条件有I/O代价
    - A8算法：有复合索引可以使用
	    - 直接使用组合索引（如`(dept_name, salary)`）进行查询，效率较高
	    - 高度更低，代价更小
    - A9算法：多个单列索引可用
	    - 算法步骤：
		    1. 对每个条件使用对应索引，得到一个记录的指针（索引项）集合
		    2. 对集合做交集操作，得到满足所有条件的记录指针集合
		    3. 读取满足条件的记录
		    4. 无索引的条件，再依次验证满足条件的记录是否满足其他条件
		- 代价：各索引查找代价+集合交集代价+读数据块；由于交集操作效率一般较高，且每个索引查找代价较低，因此总体效率通常较高
- A10：析取条件查找（$\sigma_{\theta_1\vee\theta_2}(r)$）
    - 思路：取索引项的并集，覆盖所有记录
	    - 若所有条件都有索引，对每个条件使用对应索引，得到满足条件的记录指针集合，之后取并集，读取记录
	    - 若部分条件无索引，**直接退化为线性扫描**，检查每条记录是否满足至少一个条件
	- 代价：
        - 全索引：各索引查找代价+集合并集代价+读数据块
        - 部分索引：退化为线性扫描（A1）
- A11：位图索引扫描
    - 原理：
	    - 为数据的每个页建立一个位图索引，位图中的每一位对应一个数据页，如果数据页中至少有一条记录满足条件，则该位为1，否则为0
	    - 查询时，用索引找到记录的ID，设置该页的位图位为1
	    - 线性扫描时，只读取位图索引中为1的页，减少I/O
	- 性能：匹配记录少时，接近索引扫描；匹配记录多时，接近线性扫描；适用于低基数属性（如性别、部门等）
# 排序
- 在SQL语句中，以及查询优化的过程中，都会涉及排序操作（如`ORDER BY`子句）
	- 若数据可以全部装入内存，则可以使用[[L3 排序#快速排序|快速排序]]等内排序算法，效率较高
	- 当数据大于内存时，需要用到**外存归并排序**算法进行排序
- 外存归并排序算法：
	- 思想：先分段排序，再多趟合并
		- 先创建有序段，每段大小为内存的大小，使用内排序算法对每段进行排序，并将排序结果写回磁盘
		- 之后进行多趟归并，每趟将多个有序段合并成一个更大的有序段，直到所有数据都被合并成一个有序文件
	- 算法描述：
		- 基本参数：
			- $b_r$：关系$r$存储在磁盘上的块数
			- $M$：可用内存大小（单位：页/块，也即一次最多能处理的块数）
			- $N$：阶段一结束后生成的有序段的数量
		- 阶段一：创建有序段
            1. 初始化有序段计数器`i=0`
            2. 重复以下操作，直至处理完所有数据：
	            1) 从磁盘中读取$M$块数据到内存
	            2) 使用内排序算法对这$M$块数据进行排序
	            3) 将排序后的数据写回磁盘，形成一个有序段$R_i$，并将$i$加1
	        3. 最终生成有序段数量$N=\lceil \frac{b_r}{M} \rceil$
	    - 阶段二：多路归并
            - 情况1：$N< M$，此时可以一次性将所有有序段加载到内存中进行归并，并且能预留一个块的内存用于输出
	            1. 内存分配：用$N$个内存块作输入缓冲区，读取每个有序段的第一个块；剩余一个内存块作为输出缓冲区
	            2. 循环归并：
		            - 从所有输入缓冲区中，选出当前最小的记录
		            - 将其写入输出缓冲区，若已满则将输出缓冲区写回磁盘，并清空输出缓冲区
		            - 从输入缓冲区中删除记录，若该缓冲区已空，则从对应的有序段中读取下一个块到缓冲区
                3. 直到所有有序段都被完全处理，最终得到一个完全有序的文件
            - 情况2：$N \geq M$，此时内存不够为每个有序段分配输入缓冲，因此需要每一趟归并处理$M-1$个有序段，生成新的有序段，直到最终只剩一个有序段
                - 每一趟合并后，有序段的数量将减少为$\lceil \frac{N}{M-1} \rceil$，同时每个有序段的长度变为原来的$M-1$倍
                - 重复进行多趟归并，直到最终只剩一个有序段，即完全有序的文件
                - 合并趟数：$\lceil \log_{M-1} N \rceil$，而$N=\lceil \frac{b_r}{M} \rceil$，因此总的合并趟数为$\lceil \log_{M-1}  (\frac{b_r}{M})  \rceil$
	    - 例：![[Pasted image 20260425182930.png]]
    - 代价分析：
	    1. 传输代价（Block Transfers）：读写数据块的代价
            - 阶段一：读取所有数据块进行排序，代价为$2 \times b_r$（每块数据被读取一次，写回一次）
            - 阶段二：每趟归并需要读取和写入所有数据块，合并趟数为$\lceil \log_{M-1}  (\frac{b_r}{M})  \rceil$，因此总的传输代价为$2b_r \times \lceil \log_{M-1}  (\frac{b_r}{M})  \rceil$
            - 优化：最后一趟合并的输出不需要协会磁盘，减去一个$b_r$
            - **总传输次数**：$b_r \times (2\lceil \log_{M-1}  (\frac{b_r}{M})  \rceil+1)$
            - 例：![[Pasted image 20260425183434.png]]
        2. 寻道代价（Seek Cost）：每次读写操作寻道花费的开销
            - 阶段一：每个有序段需要一次读寻道，一次写寻道，合计$2\lceil \frac{b_r}{M} \rceil$次寻道
            - 阶段二：每趟归并需要读取和写入所有数据块，合并趟数为$\lceil \log_{M-1}  (\frac{b_r}{M})  \rceil$，因此总的寻道代价为$2b_r \times \lceil \log_{M-1}  (\frac{b_r}{M})  \rceil$，再减去最后一次的写寻道
            - **总寻道次数**：$2\lceil \frac{b_r}{M} \rceil + b_r \times (2\lceil \log_{M-1}  (\frac{b_r}{M})  \rceil - 1)$

# 连接操作
- 负责将多个关系按指定条件进行拼接
- 示例：
	- `student`表：5000条记录，存储在100个块中
	- `takes`表：10000条记录，存储在400个块中
## 嵌套循环连接（Nested Loop Join，NLJ）
- 原理：类似双重`for`循环·
	- 将一个设置为**外关系（outer relation）**，另一个设置为**内关系（inner relation）**
	- 外关系的每一条元组，和内关系的每一条元组作连接条件判断
	- 满足连接条件的元组对连接过后加入结果集
- 特点：
	- 不需要索引，支持任意连接条件
	- 复杂度：$O(m  n)$，其中$m$和$n$分别是外关系和内关系的元组数，较高
- 代价分析：
	- 最坏情况：内存只能装入每个关系的1个块
		- 公式：
			- 块传输次数：$n_r \times b_s + b_r$，其中$n_r$是外关系的元组数，$b_s$和$b_r$分别是内关系和外关系的块数
			- 寻道次数：$n_r + b_r$
		- 例：若以`student`表为外关系，`takes`表为内关系，则块传输次数为$5000 \times 400 + 100 = 2,000,100$，寻道次数为$5000 + 100 = 5100$
    - 最好情况：内存足够装入内关系的所有块
        - 公式：
            - 块传输次数：$b_r + b_s$，其中$b_r$和$b_s$分别是外关系和内关系的块数
            - 寻道次数：$b_r + n_r$
        - 例：块传输次数为$100 + 400 = 500$，寻道次数为2次（顺序扫描两个关系即可）
## 块嵌套循环连接（Block Nested Loop Join，BNLJ）
- 原理：将外关系分块处理，每次将一个块加载到内存，改为按块循环
	- 外关系按磁盘块读入内存，每读入一块外关系，就扫描全体内关系块一次
	- 外关系元组和内关系元组作连接条件判断
- 伪代码：
	```text
	for each block B_r of r do begin 
		for each block B_s of s do begin 
			for each tuple t_r in B_r do begin 
				for each tuple t_s in B_s do begin 
					Check if (t_r, t_s) satisfies the join condition 
					if they do, add t_r · t_s to the result. 
				end 
			end 
		end 
	end
	```
- 代价分析：
	- 最坏情况：内存只能装入每个关系的1个块
        - 块传输次数：$b_r \times b_s + b_r$，其中$b_r$和$b_s$分别是外关系和内关系的块数
        - 寻道次数：$2*b_r$
        - 例：以`student`表为外关系，`takes`表为内关系，则块传输次数为$100 \times 400 + 100 = 40,100$，寻道次数为$2 \times 100 = 200$
    - 最好情况：内存足够装入内关系的所有块
	    - 代价和NLJ相同：块传输次数为$100 + 400 = 500$，寻道次数为2次
- 优化：利用内存大小$M$（块）进行优化：
	- 内存中预留$M-2$个块给外关系（一个块用于内关系读入，另一个块给结果输出）
	- 外关系的扫描次数从$b_r$减少到$\lceil \frac{b_r}{M-2} \rceil$，代价大幅降低
	- 代价公式：
		- 块传输次数：$\lceil \frac{b_r}{M-2} \rceil \times b_s + b_r$，其中$b_r$和$b_s$分别是外关系和内关系的块数
        - 寻道次数：$2 \times \lceil \frac{b_r}{M-2} \rceil$，因为每次扫描外关系块都需要寻道
- 特点：
	- 实现简单，无需索引，性能较好
	- 最坏情况下效率仍然较低
