- 已知用户程序的用户故事，画出数据库程序的实体-联系图，并直接依据该图设计出数据库的关系模式
- 重点是业务需求的概念设计，而不是逻辑设计
# E-R模型基础
## 实体与实体集
- 实体：业务中存在的、可区分的对象。如：
	- 具体的人：学生张三，李老师……
	- 事务：课程、项目……
	- **特点**：每个实体和其他的实体不一样，具有唯一标识
- 实体集：同一类型，具有相同属性的实体的集合
    - 例：![[Pasted image 20260608205158.png]]
- 实体的描述：实体靠属性描述，属性即实体的特征
	- `instructor`的属性：`ID`（工号）、`name`（姓名）、`dept_name`（所在院系）等
	- `student`的属性：`ID`（学号）、`name`（姓名）、`dept_name`（所在院系）等
- 实体集的符号表示：
	- 在ER图中，用矩形表示实体集，将属性写在矩形里，用下划线标记主键
	- 例：![[Pasted image 20260608205549.png]]
## 联系与联系集
- 联系：多个实体之间的关联。如:
	- 学生`44552(Peltier)`与老师`22222(Shay)`之间的联系是`advisor`（指导关系）
	- 课程`CS101`与课程`CS100`之间是`prereq`（先修）关系
- 联系集：同一类型的联系的集合，$n\geq2$个实体间的数学关系
	- 形式化定义：$R\subseteq E_1\times E_2\times \cdots \times E_n$，其中$E_i$是实体集
    - 例：`advisor`联系集![[Pasted image 20260608205817.png]]
	- 联系集的属性：联系集也可以有属性，如`advisor`联系集的属性可以是`start_date`（指导开始日期）![[Pasted image 20260608205849.png]]
	- 联系集的分类：
		- 二元联系：只有两个实体集参与的联系，如上述的`advisor`联系
			- 在ER图中，用菱形表示联系集，用直线连接参与的实体集
				- 例：![[Pasted image 20260608210005.png]]
		- 多元联系：三个或更多实体集参与的联系，如学生、课程和学期之间的`takes`联系
            - 同样用菱形表示联系集
                - 例：![[Pasted image 20260608210040.png|193]]
        - 自联系（递归联系）：同一实体集内参与两次的联系
	        - 例：![[Pasted image 20260608210139.png]]`course`实体集在联系中有两个角色：`course_id`主课程和`prereq_id`先修课程
	        - 在ER图中，为区分不同角色，会在连线上标上对应的角色名
## 联系集的约束
- 联系集必须符合业务的规则，这些规则被称为联系集的约束。如：
    - 每个学生只能有一个指导老师（`advisor`联系集的约束）
    - 每门课程必须至少有一个先修课程（`prereq`联系集的约束）
- **映射基数**（Mapping Cardinality）：描述联系集中实体之间的数量关系
    - 一对一（1:1，One-to-One）：一个实体对应另一个实体
	    - 例：一个学生只能有一个学号![[Pasted image 20260609142617.png]]
    - 一对多（1:N，One-to-Many）：一个实体A对应多个实体B，但一个实体B只能关联一个实体A
	    - 例：一个老师可以指导多个学生，但一个学生只能有一个指导老师![[Pasted image 20260609142703.png]]
    - 多对一（N:1，Many-to-One）：多个实体A对应一个实体B，但一个实体B只能关联一个实体A
        - 例：多个学生可以选修同一门课程，但一门课程只能有一个教师![[Pasted image 20260609143014.png]]
    - 多对多（M:N，Many-to-Many）：多个实体A对应多个实体B
	    - 例：多个学生可以选修多门课程，而每门课程也可以有多个学生选修![[Pasted image 20260609143046.png]]
	- 映射基数的ER图符号表示：用连线是否带箭头区分
        - 1:1联系：两端都有箭头
        - 1:N联系：从"N"端指向"1"端有箭头
        - M:N联系：两端都没有箭头
        - 示意图：![[Pasted image 20260609143308.png]]
- **参与约束**（Participation Constraints）：描述联系集中实体的参与程度
    - 全参与（Total Participation）：实体集中的每个实体都必须参与至少一次联系
        - 例：每个学生必须有一个指导老师![[Pasted image 20260609143405.png]]
    - 部分参与（Partial Participation）：联系集中的某些实体可以不参与任何联系
        - 例：有的老师可能没有指导学生![[Pasted image 20260609143432.png]]
    - 参与约束的ER图符号表示：用连线是否带双线区分
        - 全参与：连线为双线
        - 部分参与：连线为单线
        - 示意图：![[Pasted image 20260609143449.png]]
- 约束统一表示法：用连线上的`[xx..xx]`表示约束
	- 左侧数字表示最少参与次数（也即参与约束），0表示部分参与，1表示全参与
	- 右侧数字表示最多参与次数（也即映射基数），1表示一次，\*表示多次
	- 例：![[Pasted image 20260609154945.png]]
		- `instructor`是`0..*`，表示一个导师最少带0个学生（部分参与），最多带多个学生（一对多）
		- `student`是`1..1`，表示一个学生必须有一个导师（全参与），且最多只能有一个导师（一对多）
## 属性
- 属性：实体或联系的特征，如`instructor`的属性有`ID`、`name`、`dept_name`等
- 简单属性与组合属性：
	- 简单属性：不可再分的属性，如`ID`，`salary`等
	- 组合属性：可以再分成多个子属性的属性，如：
		- `name`可以分成`first_name`和`last_name`
		- `address`可以分成`street`、`city`、`state`等
	- 组合属性可以让查询更加灵活，支持单独查询某一子属性
- 单值属性与多值属性：
    - 单值属性：每个实体只能有一个值的属性，如`ID`，`name`等
    - 多值属性：每个实体可以有多个值的属性，如：
        - `phone_number`，一个人可能有多个电话号码
        - `email`，一个人可能有多个邮箱地址
    - 多值属性在ER图中用方括号`[]`表示![[Pasted image 20260609155323.png]]
- 派生属性：可以通过其他属性计算得到的属性，如：
    - `age`可以通过`birth_date`计算得到
    - `total_credits`可以通过学生选修的课程学分计算得到
    - 派生属性在ER图中用圆括号`()`表示，详见上例
- 域（Domain）：属性的取值范围，如：
    - `ID`的域可以是整数
    - `name`的域可以是字符串
    - 域的定义有助于数据的验证和约束，确保数据的正确性和一致性
## 码/键
- 键：
	- [[L1 关系模型#^4f884c|超键]]
	- [[L1 关系模型#^449f65|候选键]]
	- [[L1 关系模型#^697942|主键]]
- 码：
	- 实体集的主码：唯一标识实体集中的每个实体的属性或属性组合
		- 主码一定是候选键中的一个
		- 在ER图中，主码用下划线标记
	- 联系集的主码：
		- 参与联系的实体集的主码的**并集**
		- 如果联系集有属性，则主码还需要包括联系集的属性
		- 主码的选择取决于映射基数：
			- 一对一：可以选择任一实体集的主码作为联系集的主码
			- 一对多：必须选择"多"端实体集的主码作为联系集的主码
			- 多对多：必须选择双方主码的组合作为联系集的主码
# 弱实体集
- 场景：有些实体无法独立存在，必须依赖于其他实体才能被唯一标识，这些实体称为弱实体（Weak Entity）
	- 例：某门课程`course`可以开设多个教学班级`section`，只依靠教学班级自身的`sec_id`、`semester`、`year`等属性无法唯一标识一个教学班级，因为不同的课程可能有相同的教学班级信息；因此，必须结合课程的`course_id`才能唯一标识一个教学班级
- 基本概念：
	- 弱实体集：依赖于另一个**标识性实体**（Identifying Entity）的实体集
		- 弱实体集自身的属性无法凑出完整的主键，必须结合标识实体的主键，才能唯一确定实体
	- 分辨符（Discriminator）：弱实体集中的一个属性或属性组合，用于区分同一标识实体下的不同弱实体
        - 例：在上述的`section`弱实体集中，`sec_id`、`semester`、`year`可以作为分辨符，因为它们可以区分同一课程下的不同教学班级
    - 标识性联系（Identifying Relationship）：连接弱实体集和标识性实体集的联系
	    - 弱实体在标识性联系中一定是全参与的（因为其必须依赖于标识实体）
	    - 通常是一对多关系：一个标识实体可以对应多个弱实体，但一个弱实体只能对应一个标识实体
	- 强实体集：非弱实体集，即可以独立存在的实体集
- ER图表示：![[Pasted image 20260610103537.png]]
    - 弱实体集用双矩形表示
    - 标识性联系用双菱形表示
    - 分辨符属性用下划虚线表示
    - 标识性联系的连线为双线，表示弱实体的全参与约束
    - 标识性强实体用单矩形表示，下划线注出其主码属性
- 注：
	- 弱实体集可以参与标识联系之外的其他联系
	- 不建议把标识实体的主键直接加到弱实体的属性中，不利于业务逻辑的清晰表达和维护
	- 一个弱实体可以有多个标识实体，通过多个标识性联系连接
# ER图
- ER图的规范：

	|元素类型|绘制规范|适用场景|
	|---|---|---|
	|普通强实体|单线矩形，主键属性加**实线下划线**|可依靠自身属性独立唯一标识的实体，如`department`、`student`|
	|弱实体|双线矩形，分辨符（部分键）加**虚线下划线**|存在依赖其他实体、无法独立确定主键的实体，如案例里的`section`教学班|
	|普通联系|单线菱形，连线单线 = 部分参与、双线 = 全参与|实体间常规业务关联|
	|标识性联系（弱实体专属）|双线菱形，弱实体侧必须用双线连接（强制全参与）|绑定弱实体和它依附的标识强实体|
	|属性|椭圆框，用直线连所属实体 / 联系|实体的特征，或是联系自带的特征（如选课的`grade`成绩）|
	|递归自联系|同一个实体矩形引出两条线，在连线上标注不同**角色名**|实体内部的关联，如课程的先修课`prereq`|
- ER图的设计：以大学数据库为例
	1. 识别所有强实体，属性和主键
		- `department`院系：主键`dept_name`（院系名全局唯一），属性`building`所在楼栋、`budget`预算
		- `course`课程：主键`course_id`，属性`title`课程名、`credits`学分；自带递归自联系`prereq`（先修课），用`course_id`（主课程）、`prereq_id`（先修课程）两个角色区分
		- `instructor`教师：主键`ID`，属性`name`姓名、`salary`薪资；归属某个院系
		- `student`学生：主键`ID`，属性`name`姓名、`tot_cred`总已修学分；主修某个院系
	2. 设计实体间的二元联系：
		- `inst_dept`：教师 - 院系的多对一联系（一个院系有多名教师，一名教师仅属于一个院系）
		- `stud_dept`：学生 - 院系的多对一联系（一个院系有多名学生，一名学生仅主修一个院系）
		- `course_dept`：课程 - 院系的多对一联系（一个院系开设多门课程，一门课程归属一个院系）
		- `advisor`：学生 - 导师的一对多联系：学生侧双线全参与（每个学生必须有 1 个导师），教师侧单线部分参与（部分教师不承担带学生的工作）
	3. 扩展业务需求，补充实体和关联：
		- 新增实体：
			- `classroom`教室：复合主键`building+room_number`（楼栋 + 房间号唯一确定教室），属性`capacity`容纳人数
			- `section`教学班：弱实体，依附`course`课程；分辨符为`sec_id+semester+year`，通过双线菱形`sec_course`（标识性联系）和课程绑定
			- `time_slot`上课时段：主键`time_slot_id`，属性`day`星期、`start_time`开始时间、`end_time`结束时间
		- 新增业务联系：
			- `teaches`：教师 - 教学班的授课多对多联系（一名教师可教多个班，一个班可由多名教师合授）
			- `takes`：学生 - 教学班的选课多对多联系，自带专属属性`grade`（最终成绩，用椭圆连接菱形）
			- `sec_class`：教学班 - 教室的多对一联系（一个教学班固定一间教室，一间教室不同时段可安排多个班）
			- `sec_time_slot`：教学班 - 上课时段的关联联系，绑定班级的上课时间安排

# ER图转化关系模式
- 基本规则：
	- 每一个实体集/联系集都对应一个专属命名的关系模式（数据库表）
	- 模式的每一列对应一个属性，所有的列名在表内唯一
	- 所有的主键和约束转化为表的约束
- 强实体集的转换：
	- 无复杂属性的强实体：直接平移为关系模式，将实体主键直接作为表的主键
	- 带复杂属性的强实体：
		- 组合属性：将组合属性的每个子属性作为关系模式的一列，组合属性本身不作为列
		- 派生属性：派生属性不转化为关系模式的一列，因为它可以通过其他属性计算得到
		- 多值属性：实体`E`的多值属性`M`建立对应的`EM`表处理
			- 每一列对应实体`E`的主键和多值属性`M`本身
			- `EM`的联合主键为`E`的主键和`M`，确保每个实体的每个多值属性值唯一
			- 例：多值属性`phone_number`转化为`inst_phone`表，包含`ID`（教师主键）和`phone_number`两列，联合主键为`(ID, phone_number)`
- 不同基数联系集的转换：
	- 多对多联系：新建单独的表
		- 表的列为两端参与实体的主键和联系自身的属性
		- 表的联合主键为两端参与实体的主键组合
		- 例：`advisor`师生多对多联系，转化成`advisor(s_id, i_id)`表，包含学生ID和教师ID两列，联合主键为`(s_id, i_id)`
	- 多对一/一对多联系：不需要单独键联系表
		- 将"一"端的实体主键作为外键，加到"多"端的实体表中
		- 例：`instructor`与`department`的多对一联系`inst_dept`，将院系主键`dept_name`作为外键加到教师表`instructor`中，表示教师所属院系
		- **参与约束的影响**：如果"多"端是部分参与，则新增的外键列有可能出现`NULL`值
	- 一对一联系：可以选择任意一侧的实体表，将对方的主键作为外键加入表中
- 弱实体集的转换：
    - 弱实体集单独生成一张表，表的列包括：
	    - 依附的标识强实体的主键
	    - 弱实体自身的分辨符（部分键）
	    - 例：上述弱实体集`section`转化为`section(course_id, sec_id, semester, year)`表，包含课程ID（标识强实体主键）和分辨符属性`sec_id`、`semester`、`year`
	- 标识性联系的处理：
        - 标识性联系不单独生成表，而是通过弱实体表中的外键来体现
        - 例：`section`表中的`course_id`既是标识强实体`course`的主键，也是标识性联系`sec_course`的体现，不需要单独建表
# ER模型进阶
- 冗余属性：有的实体属性在建模成关系模型时，会出现保存相同业务语义的冗余信息
	- 例：![[Pasted image 20260610115150.png]]此处`member`联系集和`instructor`实体本身都存储了`dept_name`属性，导致冗余
	- 冗余属性的影响：数据冗余会导致数据不一致，更新时可能发生异常，维护成本较高
	- 处理：仅保留一种属性，优先选择用外键的方法表示联系，而不是使用冗余的联系集
- 非二元联系集：有些联系必须设计多元联系（如某学生在某教师指导下参与某项目）才能表达完整语义
	- 约束规则：多元联系集必须遵循特殊的基数约束规范
		- 多元联系最多只能有一个箭头（指向实体），表示多对一关系
		- 例：![[Pasted image 20260610115520.png]]
			- 若箭头指向`project`，表示"一个学生和一个导师的组合只能参与一个项目"
			- 若箭头指向`student`，表示"一个导师和一个项目的组合只能指导一个学生"
- 精化与泛化：
	- 精化（Specialization）：从一个高层次的通用实体，根据特征差异，划分出多个低层次的子实体。子实体**继承**父实体的属性和联系，并且可以有自己的特有属性和联系
        - 例：从`person`实体精化出`student`和`instructor`两个子实体，`person`具有属性`ID`、`name`、`dept_name`，而`student`特有属性`tot_cred`，`instructor`特有属性`salary`
        - 表示：![[Pasted image 20260610120931.png]]用空心箭头指向父实体
	    - 精化的分类：
		    - 不相交（Disjoint）：一个实体只能属于一个子实体，子实体之间没有重叠
		    - 重叠（Overlapping）：一个实体可以同时属于多个子实体，子实体之间存在重叠
        - 完整性约束：
	        - 全参与（Total Participation）：父实体集中的每个实体必须至少属于一个子实体
	        - 部分参与（Partial Participation）：父实体集中的某些实体可以不属于任何子实体
	    - 精化关系转化为表：
		    - 垂直分割：将公共属性放到父表，特有属性放到子表，通过主键相互关联
			    - 例：![[Pasted image 20260610121305.png|194]]
			    - 没有数据冗余，但查询时需要连接父表和子表
			- 水平分割：每个子实体单独建表，表中包含父实体的所有属性和自身特有的属性
				- 例：![[Pasted image 20260610121455.png]]
				- 有数据冗余，但查询时不需要连接父表和子表
	- 泛化（Generalization）：有多个共同特征的实体抽象出更高层级的父实体的过程，和精化过程相反
		- 表示方法和精化相同
- 聚合（Aggregation）：当联系集本身需要参与其他联系集时，需要聚合解决数据冗余
	- 例：已有三元联系`proj_guide`，表示学生-导师-项目关联，现需要增加`eval_for`联系，给每个学生-导师-项目组合一个评估结果`evaluation`，如果直接把`eval_for`联系连接到`student`、`instructor`、`project`三个实体上，会导致数据冗余
	- 解决方法：将`proj_guide`联系集聚合成一个**虚拟实体**`student_instructor_project`，然后让`eval_for`联系连接到这个抽象实体上
		- 示意图：![[Pasted image 20260610130353.png]]
		- 关系模式：直接将聚合的联系集的主键作为新联系表的外键
			- 例：`eval_for(s_id, i_id, p_id, evaluation)`表，其中`s_id`、`i_id`、`p_id`是`proj_guide`联系集的主键，也是新表的外键，`evaluation`是新联系集的属性
- UML图与ER图的对应：
	- |ER 模型元素|UML 类图对应元素|说明|
		|---|---|---|
		|实体集（矩形）|类（Class，矩形）|UML 类图的矩形分三层：类名、属性、方法；ER 里的实体只有属性，UML 可加方法（如数据库操作）|
		|简单 / 组合 / 多值 / 派生属性|类的属性|UML 属性可加访问修饰符：`+public`、`-private`、`#protected`|
		|二元联系（菱形）|关联（Association，直线）|用直线连接两个类，连线上标角色名和多重度|
		|联系的属性|关联类（Association Class）|在关联线上挂一个类，存放联系的属性|
		|基数约束（1:1/1:N/M:N）|多重度（Multiplicity）|如`0..1`、`1..*`、`*`，注意 UML 里的多重度位置和 ER 里的箭头方向相反|
		|三元联系（菱形）|关联（可用关联类表示）|UML 支持 n 元关联，或用关联类处理带属性的三元联系|
		|精化 / 泛化（ISA 箭头）|继承（Generalization，空心三角箭头）|从子类指向父类，和 ER 里的 ISA 箭头方向一致|
