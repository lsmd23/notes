- 可串行化的事务并发执行，需要并发控制才能确保事务执行结果的正确性
# 基于锁的控制协议
- [[L7 事务#隔离级别|事务的隔离级别]]
- 锁：锁是控制并发访问的机制
	- 共享锁（S锁/读锁）：允许多个事务同时读取数据项，但不允许任何事务修改数据项
		- 可以以`lock-S`相关指令申请
	- 排他锁（X锁/写锁）：允许一个事务修改数据项，并且不允许任何其他事务读取或修改该数据项
		- 可以以`lock-X`相关指令申请
	- 锁相容性矩阵：
		- |已持有锁|申请 S 锁|申请 X 锁|
			|---|---|---|
			|S 锁|✅ 兼容|❌ 不兼容|
			|X 锁|❌ 不兼容|❌ 不兼容|
	- 锁的实现：
		- 读锁：![[Pasted image 20260528195352.png]]
			- 如果事务持有锁，直接读
			- 如果没有锁，等待直到没有其他事务持有写锁，加锁后读
		- 写锁：![[Pasted image 20260528195459.png]]
			- 如果事务持有X锁，直接写
			- 如果持有S锁，升级为X锁后写
			- 如果没有锁，等待直到没有其他事务持有任何锁，加锁后写
- 锁协议：简单的加锁/去锁是不行的，必须保证事务在执行过程中遵守一定的锁协议，才能确保并发执行的正确性
    - 两段锁协议（Two-Phase Locking, 2PL）：
	    - 定义：事务必须遵守以下两个阶段的规则：
	        1. **第一阶段（Growing Phase）**：事务可以申请锁，但不能释放任何锁
	        2. **第二阶段（Shrinking Phase）**：事务可以释放锁，但不能申请任何新锁
	    - 串行化保证：遵守两段锁协议的调度一定是冲突可串行化的
	    - 问题：2PL协议会有以下问题：
		    - 死锁（Deadlock）：当两个或多个事务相互等待对方释放锁时，就会发生死锁，导致这些事务无法继续执行
			    - 例：![[Pasted image 20260528194510.png]]
			    - 解决：可以回滚一个事务来打破死锁，或者使用[[L8 并发控制#死锁处理|死锁处理]]
			    - 饥饿（Starvation）：如果一个事务一直无法获得所需的锁，因为其他事务不断地占用这些锁，那么这个事务就会一直等待，导致饥饿问题
			- 级联回滚（Cascading Rollback）：如果涉及锁有串联使用，当一个事务回滚时，可能会导致其他事务也需要回滚，形成级联回滚
				- 例：![[Pasted image 20260528194724.png]]如果事务T10回滚，那么事务T11、T12也必须回滚，导致级联回滚
    - 严格两段锁协议（Strict Two-Phase Locking, Strict 2PL）：
        - 事务的**所有排他锁（X锁）**，必须等到事务提交/回滚之后，才能进行释放
        - 这样，其他事务永远无法读取未提交的数据，避免了级联回滚的问题
	        - 例：![[Pasted image 20260528194927.png]]
	- 严厉两段锁协议（Rigorous Two-Phase Locking, Rigorous 2PL）：
		- 事务的**所有锁（包括共享锁和排他锁）**，必须等到事务提交/回滚之后，才能进行释放
		- 事务以串行化顺序执行，提供更强的隔离级别，但性能较低
			- 例：![[Pasted image 20260528195010.png]]
- 锁的管理：
	- 锁转换：事务可以在不同阶段，对锁的模式进行转换
		- 锁升级（S锁转X锁）：当一个事务需要修改一个数据项时，可以将已经持有的S锁升级为X锁
			- 这一操作只能在增长阶段进行
		- 锁降级（X锁转S锁）：当一个事务完成了对一个数据项的修改后，可以将已经持有的X锁降级为S锁
			- 这一操作只能在缩减阶段进行
	- 锁管理器：数据库系统中的一个组件，负责管理锁的申请、释放和转换，有如下功能：
		- 维护**锁表**：记录每个数据项当前持有的锁类型和持有锁的事务列表
		- 处理锁请求：当一个事务申请锁时，根据相容性矩阵决定是否授予锁
		- 如果锁冲突，事务进入等待状态，并将其加入等待队列
		- 如果死锁，检测死锁并回滚处理
    - 锁表（Lock Table）：一个哈希表，每个数据项对应一个请求队列
	    - 示意图：![[Pasted image 20260528195821.png]]
	    - 加锁时，新请求加到队尾，检查兼容性，若不兼容，进入等待状态
	    - 解锁时，删除锁请求，检查后续队列中等待的请求是否可以被满足，满足则授予锁并唤醒相应事务
	    - 事务中止时，删除该事务的所有锁请求，检查后续队列中等待的请求是否可以被满足，满足则授予锁并唤醒相应事务
# 死锁处理
* 死锁：当系统中存在一组事务，组内每个事务都在等待组内另一个事务释放锁时，系统就处于死锁状态。形式化地，存在一组事务$T_0​→T_1​→T_2​→...→T_{n−1}​→T_{n}​→T_0$​，形成了一个循环等待链，没有任何事务能继续推进
* **死锁预防**：破坏死锁的产生条件从而预防死锁的发生
    - 一次性加锁（预声明）：
		- 事务在开始执行前，必须一次性申请全部现需要的锁，不能只获取一部分需要的锁
		- 原理：事务不会在持有锁的同时申请新的锁，因此不会形成循环等待
		- 特点：实现简单，但并发度低下
	- 基于图的死锁预防：
		- 对所有数据项按照某种关系排序，事务必须按照这个顺序申请锁
		- 原理：破坏循环等待条件
		- 特点：并发度相对较高，但修改数据项时需要调整排序，增加了系统的复杂性
	- **基于时间戳的死锁预防**：利用事务的时间戳决定等待的事务和回滚的事务
		- 非抢占式（Wait-Die）实现：时间戳靠后的事务不等待时间戳靠前的事务，而是直接回滚自己
			- 当事务$T$请求事务$S$持有的锁时，若$T$的时间戳早于$S$，则$T$等待；否则，$T$回滚
			- 特点：减少了不必要回滚，但时间戳靠后的事务可能多次回滚重启
		- 抢占式（Wound-Wait）实现：时间戳靠前的事务不等待时间戳靠后的事务，而是直接回滚时间戳靠后的事务，从而获取锁
			- 当事务$T$请求事务$S$持有的锁时，若$T$的时间戳早于$S$，则$S$回滚，将锁交由$T$；否则，$T$等待
			- 特点：减少了不必要回滚，但时间戳靠前的事务可能频繁回滚其他事务、
	- **基于超时的死锁预防**：
		- 事务申请锁时，只等待一个固定的时间，如果超过这个时间仍未获得锁，则认为发生了死锁，回滚该事务
		- 特点：实现简单，但可能误判非死锁情况为死锁，导致不必要的回滚甚至饥饿现象
- 死锁检测与恢复：允许死锁发生，但通过检测恢复算法解决死锁问题
	- **基于等待图（Wait-for Graph）的死锁检测**：用等待图描述事务间的等待关系，通过检测环的存在判断死锁
		- 等待图：$G=(V,E)$，其中$V$是事务集合，$E$是等待关系的集合，如果事务$T_i$等待事务$T_j$释放锁，则在图中有一条从$T_i$到$T_j$的有向边
			- 例：![[Pasted image 20260528203616.png]]存在环，说明存在死锁
		- 检测时机：
			- 定期检测：系统每隔一段时间执行一次死锁检测算法，检查等待图中是否存在环
			- 事件触发检测：当一个事务进入等待状态时，立即执行死锁检测算法，检查等待图中是否存在环
	- 死锁恢复：检测到死锁后，必须回滚事务释放锁
		- 回滚事务的选取：选取代价最小的事务回滚，事务执行的时间，事务修改的数据量，事务优先级都可以作为选取回滚事务的依据
		- 饥饿问题：如果总是回滚同一个事务，可能导致该事务一直无法完成，形成饥饿现象
			- 将回滚次数也计入代价计算，避免频繁回滚同一个事务
# 多粒度封锁
- 数据库的粒度：
	- 示意图：![[Pasted image 20260528204300.png]]数据库的不同区域管理的大小是不同的，如果对整个数据库使用相同粒度的锁，会导致不必要的锁冲突和性能下降
		- 粗粒度锁开销低，但并发度差；细粒度锁并发度高，但开销大
		- 多粒度封锁：允许事务在不同粒度上申请锁，根据需要选择合适的粒度进行锁定，从而提高并发性能
			- 对单条记录的读写，可以使用记录级别的细粒度锁；对整个表的操作，可以使用表级别的粗粒度锁；对整个数据库的维护，可以使用数据库级别的锁
- 意向锁（Intention Lock）：
	- 为解决多粒度锁判断冲突时对数据库结构树的遍历问题，引入意向锁，提前预告事务的加锁意图从而判断冲突
	- 锁模式：
		- IS（Intention Shared Lock）：表示事务打算对树的下层节点加共享锁
		- IX（Intention Exclusive Lock）：表示事务打算对树的下层节点加排他锁
		- SIX（Shared and Intention Exclusive Lock）：事务对当前节点加S锁，同时打算对树的下层节点加X锁
		- 意向锁的兼容性矩阵：
			- 
				|已持有锁|申请 IS|申请 IX|申请 S|申请 SIX|申请 X|
				|---|---|---|---|---|---|
				|**IS**|✅兼容|✅兼容|✅兼容|✅兼容|❌不兼容|
				|**IX**|✅兼容|✅兼容|❌不兼容|❌不兼容|❌不兼容|
				|**S**|✅兼容|❌不兼容|✅兼容|❌不兼容|❌不兼容|
				|**SIX**|✅兼容|❌不兼容|❌不兼容|❌不兼容|❌不兼容|
				|**X**|❌不兼容|❌不兼容|❌不兼容|❌不兼容|❌不兼容|
				- **IS 和所有非 X 锁兼容**：IS 只是表示 “要给下层加 S 锁”，不会影响其他事务的意向锁或 S 锁，只有 X 锁会冲突（因为 X 锁会隐式给所有子节点加 X 锁，和下层的 S 锁冲突）；
				- **IX 和 S/SIX 不兼容**：IX 表示 “要给下层加 X 锁”，而 S/SIX 表示当前节点加了 S 锁（隐式给所有子节点加 S 锁），X 和 S 是不兼容的，所以 IX 和 S/SIX 冲突；
				- **SIX 仅和 IS 兼容**：SIX 表示当前节点加了 S 锁（隐式给所有子节点加 S 锁）同时要给下层加 X 锁，而 IS 只是表示要给下层加 S 锁，不会影响当前节点的 S 锁，所以 SIX 和 IS 兼容；而 IX 和 X 都表示要给下层加 X 锁，会和 SIX 的 S 锁冲突，所以不兼容；
- 多粒度封锁协议：
	- 内容：
		1. 必须遵守相容性矩阵
		2. 必须先锁定根节点
		3. 加S/IS锁的前提：事务要给某个节点加S锁，必须先在该节点的所有祖先节点上加IS/IX锁
		4. 加X/IX/SIX锁的前提：事务要给某个节点加X锁，必须先在该节点的所有祖先节点上加IX/SIX锁
		5. 必须遵循两阶段锁协议
		6. 解锁必须从叶节点开始
	- 性质：
		- 加锁的顺序是数据库结构树的自顶向下的顺序，解锁的顺序是自底向上的顺序
		- 锁粒度升级：若某个层级的锁数量很多，可以将多个细粒度锁升级为一个粗粒度锁，减少锁的数量和管理开销
# 插入与删除操作
- 插入与删除的加锁规则：
	- 删除操作：事务要删除一条元组，必须先对该元组加X锁，否则不允许删除
		- 删除操作会导致数据项的删除，因此必须加X锁以保证数据的一致性
	- 插入操作：事务要插入一条新元组，必须对这条新元组加X锁
		- 插入操作会导致数据项的增加，同样需要加X锁以保证数据的一致性、
- 幻读问题：
	- 插入操作导致的幻读：当一个事务在执行过程中，另一个事务插入了满足查询条件的新记录，导致前者两次查询的结果不同异常
	- 更新操作导致的幻读：当一个事务在执行过程中，另一个事务更新了满足查询条件的记录，使得前者两次查询的结果不同异常
	- 幻读的消除：
		- 幻读的本质是因为锁的对象是一个数据集合，只能解决元组级别的冲突，幻读冲突则是谓词界别的冲突，而不是某一个固定的数据项
		- 解决：
			- 对关系表关联一个数据项，表示“表中包含的元组”这一信息
			- 谓词读事务在查询时，对这个数据项加S锁，表示它需要知道表中包含的元组信息
			- 增删改事务在修改表中元组时，对这个数据项加X锁，表示它修改了表中包含的元组信息
			- 本质是表级锁，并发度较低
# 基于时间戳的并发控制协议
- 时间戳协议：利用事务的时间戳，定义执行顺序，从而保证调度的可串行化
	- 事务的时间戳：每个事务$T_i$进入系统时，会被分配唯一的时间戳$TS(T_i)$
		- 单调递增的一组数据，后进入的事务，时间戳更大
	- 数据项的时间戳：每个数据项$Q$维护两个时间戳：
		- 读时间戳$RTS(Q)$：所有成功执行过的`read(Q)`事务中，最大的时间戳（也即最近一次成功读取$Q$的事务的时间戳）
		- 写时间戳$WTS(Q)$：所有成功执行过的`write(Q)`事务中，最大的时间戳（也即最近一次成功写入$Q$的事务的时间戳）
- 时间戳协议下的读写操作：
	- 读操作`read(Q)`：
		1. 如果$TS(T_i) < WTS(Q)$，则$T_i$读取了一个被后续事务修改过的数据项，违反了时间戳顺序，必须回滚$T_i$
		2. 否则，$T_i$成功读取$Q$，更新$RTS(Q) = max(RTS(Q), TS(T_i))$
	- 写操作`write(Q)`：
		1. 如果$TS(T_i) < RTS(Q)$，则$T_i$写入了一个被后续事务读取过的数据项，违反了时间戳顺序，必须回滚$T_i$
		2. 如果$TS(T_i) < WTS(Q)$，则$T_i$写入了一个被后续事务修改过的数据项，违反了时间戳顺序，必须回滚$T_i$
		3. 否则，$T_i$成功写入$Q$，更新$WTS(Q) = TS(T_i)$
	- 例：![[Pasted image 20260529171329.png]]
	- 优化方案：Thomas写规则
		- 当写操作`write(Q)`满足$TS(T_i) < WTS(Q)$时，基础协议需要回滚$T_i$，但实际上$T_i$写入的数据已经被后续事务覆盖了，不会对系统造成任何影响
		- 因此，可以优化这一次写操作，直接**丢弃$T_i$的写入**，而不是回滚$T_i$，从而提高系统的性能
# 基于验证的并发控制协议
- 一种乐观的并发控制协议，允许事务在执行过程中不加锁，但在提交时进行验证，确保事务的执行结果是正确的
- 事务执行的三个阶段：每个事务$T_i$按照三个阶段执行：
	1. 读执行阶段（Read and execution phase）：$T_i$执行所有的读操作，将数据写入临时的本地变量中，不修改数据库中的数据项
	    - 这一阶段可以并发执行而没有数据一致性问题 
	2. 验证阶段（Validation phase）：当$T_i$准备提交时，进入验证阶段，检查$T_i$的执行是否与其他事务冲突
		- 这一阶段必须串行执行，以确保验证的正确性
    3. 写回阶段（Write phase）：如果验证通过，$T_i$将本地变量中的数据写回数据库中，完成提交；如果验证失败，$T_i$必须回滚
- 基于时间戳的有效性验证：
	- 每个事务$T_i$记录三个时间戳：
		- 开始时间戳$Start(T_i)$：事务$T_i$开始执行的时间
		- 验证时间戳$Validation(T_i)$：事务$T_i$进入验证阶段的时间
		- 完成时间戳$Finish(T_i)$：事务$T_i$完成写阶段的时间
		- 事务的时间戳$TS(T_i)=Validation(T_i)$，也即事务的串行化顺序是由进入验证阶段的时间决定的
	- 有效性检查：检查当前事务$T_j$，以及所有比它早的事务$T_i$，满足以下任一条件，则事务不冲突：
		1. $Finish(T_i) < Start(T_j)$：$T_i$在$T_j$开始之前完成，二者没有重叠，不冲突
		2. $Start(T_j)<Finish(T_i) < Validation(T_j)$，且$T_i$的写集合与$T_j$的读集合没有交集：$T_i$在$T_j$的验证阶段之前完成，二者有重叠，但没有数据冲突

# 多版本并发控制
- 思路：为每个数据维护多个版本，使得读和写操作不再是完全冲突的，从而大幅提高并行度
	- 目前现代数据库的主流并发控制协议
- 多版本数据项：
	- 每个数据项维护一个版本序列：$<Q_1,Q_2,...,Q_m>$，其中每个版本$Q_i$都包含三个字段：
		- `Content`：数据项的值
		- `W-Timestamp`：写入该版本的事务的时间戳，表示该版本是由哪个事务写入的
		- `R-Timestamp`：所有读取过该版本的事务中，最大的时间戳，表示最近一次读取该版本的事务的时间戳
	- 赋值规则：
		- 创建新版本：事务$T_i$创建新版本$Q_k$时，初始化`R-Timestamp(Q_k)`和`W-Timestamp(Q_k)`为$TS(T_i)$
		- 读取版本：事务$T_i$读取版本$Q_k$时，若$TS(T_i) > R-Timestamp(Q_k)$，则更新`R-Timestamp(Q_k) = TS(T_i)`，表示$Q_k$被$T_i$读取过了
- 多版本下的读写操作：
	- 读操作`read(Q)`：
		1. 找到数据项$Q$的所有版本中，`W-Timestamp`小于或等于$TS(T_i)$的最大版本$Q_k$
		2. 直接读取$Q_k$的`Content`，并按照读取版本的规则更新`R-Timestamp(Q_k)`
	- 写操作`write(Q)`：先按照读操作的规则，找到目标版本$Q_k$，分情况讨论：
		1. 如果$TS(T_i) < R-Timestamp(Q_k)$，则$T_i$写入了一个被后续事务读取过的数据项，违反了时间戳顺序，必须回滚$T_i$
		2. 如果$TS(T_i) = W-Timestamp(Q_k)$，则这个版本就是事务自己创建的，这次写入是同一事务的后续写入，直接更新$Q_k$的`Content`即可
		3. 否则，创建新的版本$Q_{k+1}$，设置`Content`为新的值，初始化`W-Timestamp(Q_{k+1})`和`R-Timestamp(Q_{k+1})`为$TS(T_i)$，并将$Q_{k+1}$加入版本序列中
- 多版本协议的读写完全不阻塞，且没有脏读问题，但产生了存储开销的问题，以及需要定期清理过期版本
