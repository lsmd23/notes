- 如何设计出没有信息冗余，没有数据异常，方便进行一致性检查的关系范式？
# 第一范式（1NF）
- 域的原子性：如果一个属性的域里的元素都是不可分割的单元，则这个域是原子的
	- 例：![[Pasted image 20260611153328.png]]以上的表中，`Customers`一栏包含了多个属性，是典型的非原子值
		- 非原子的域会导致数据冗余和维护上的困难
- 第一范式（1NF）：**一个关系模式$R$满足第一范式，当且仅当$R$的所有属性的域都是原子的**
	- 问题：第一范式只对域的原子性有要求，但对域的定义，设计均没有要求，会导致设计时依赖混乱，数据冗余和异常
		- 例：教师关系模式`instructor(ID, name, dept_name, salary, building, budget)`
			- 信息冗余：同一院系的`building`和`budget`会重复出现
			- 修改异常：院系的`budget`发生变化，需要修改多个元组，容易遗漏
			- 插入异常：如果要插入一个新的院系，但该院系没有教师，那么无法插入该院系的信息
			- 删除异常：如果删除了某个教师的元组，可能会导致该院系的信息也被删除，造成数据丢失
		- 解决：进行模式的拆解，消除冗余和异常
# 函数依赖
- 定义：
	- 函数依赖（functional dependency）：设$R$是一个关系模式，$\alpha\subseteq R$，$\beta\subseteq R$（$\alpha$和$\beta$都是属性集），如果对于$R$的任意合法实例$r$，只要元组$t_1$，$t_2$在$\alpha$上的值相等（$t_1[\alpha]=t_2[\alpha]$），就必然在$\beta$上的值也相等（$t_1[\beta]=t_2[\beta]$），则称$\alpha$函数决定$\beta$，记为$\alpha\to\beta$，读作"$\alpha$函数依赖于$\beta$"在$R$上成立
		- 非形式化的，函数依赖指属性集$\alpha$的值可以决定属性集$\beta$的值
		    - 例：![[Pasted image 20260611160739.png]]在这个关系模式中，$A\rightarrow B$不成立；但$B\rightarrow A$成立，因为$B$的值可以唯一确定$A$的值
	- 几何意义：
		- 实例（Instance）：某个关系模式的一个具体表格，包含了若干元组（Tuple）
		- 实例空间（Instance Space）：所有可能的实例组成的集合
			- 例：![[Pasted image 20260611161832.png]]
		- 函数依赖约束子集：满足函数依赖$\alpha\to\beta$的实例空间中的实例集合
			- 例：![[Pasted image 20260611161905.png]]
	- 应用：
	    - 超键的函数依赖定义：$K$是$R$的超键，当且仅当$K\rightarrow R$成立，即$K$的值可以唯一确定关系模式$R$中的所有属性的值
	    - 候选键的函数依赖定义：$K$是$R$的候选键，当且仅当$K\rightarrow R$成立，并且对于$K$的任何真子集$K'$，$K'\not\rightarrow R$，即候选键是最小的超键
	    - 检验关系的合法性：函数依赖定义了对应实例空间中的合法子集，只有满足函数依赖约束的实例才是合法的关系实例
		    - **满足**（Satisfied）：针对**单个实例**的描述，如某个具体的表格满足函数依赖约束
			    - 例：![[Pasted image 20260611162138.png]]上述关系实例满足`loan-number → customer-name`
			- **保持**（Hold on）：针对**整个关系模式**的描述，当$R$的所有合法实例都满足依赖$\alpha\to\beta$时，称$\alpha\to\beta$在关系模式$R$上保持
	    - 函数依赖与一致性约束：函数依赖可以刻画部分属性决定另一部分属性的业务规则
		    - 例：贷款信息模式`Loan-info-schema = (customer-name, loan-number, branch-name, amount)`
			    - 成立的函数依赖：
	                - `loan-number → amount`：贷款号可以唯一确定贷款金额
	                - `loan-number → branch-name`：贷款号可以唯一确定贷款所在的分行
	            - 不成立的函数依赖：
	                - `loan-number → customer-name`：贷款号无法唯一确定客户姓名，因为一个客户可能有多个贷款
	- 平凡函数依赖（trivial functional dependency）：如果$\beta\subseteq\alpha$，则$\alpha\to\beta$是平凡函数依赖
		- 这一类函数依赖没有实际意义，因为$\alpha$的值自然包含了$\beta$的值，没有约束作用
		- 后续所有讨论的函数依赖默认是非平凡函数依赖
