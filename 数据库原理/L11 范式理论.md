# 第一范式（1NF）
- 域的原子性：如果一个属性的域里的元素都是不可分割的单元，则这个域是原子的
	- 例：![[Pasted image 20260611153328.png]]以上的表中，`Customers`一栏包含了多个属性，是典型的非原子值
		- 非原子的域会导致数据冗余和维护上的困难
- 第一范式（1NF）：**一个关系模式$R$满足第一范式，当且仅当$R$的所有属性的域都是原子的**
	- 问题：第一范式只对域的原子性有要求，但对域的定义，设计均没有要求，会导致设计时依赖混乱，数据冗余和异常
		- 例：教师关系模式`instructor(ID, name, dept_name, salary, building, budget)`
			- 信息冗余：同一院系的`building`和`budget`会重复出现
			- 修改异常：院系的`budget`发生变化，需要修改多个元组，容易遗漏
			- 插入异常：如果要插入一个新的院系，但该院系没有教师，那么无法插入该院系的信息
			- 删除异常：如果删除了某个教师的元组，可能会导致该院系的信息也被删除，造成数据丢失
		- 解决：进行模式的拆解，消除冗余和异常
# 函数依赖
- 定义：
	- 函数依赖（functional dependency）：设$R$是一个关系模式，$\alpha\subseteq R$，$\beta\subseteq R$（$\alpha$和$\beta$都是属性集），如果对于$R$的任意合法实例$r$，只要元组$t_1$，$t_2$在$\alpha$上的值相等（$t_1[\alpha]=t_2[\alpha]$），就必然在$\beta$上的值也相等（$t_1[\beta]=t_2[\beta]$），则称$\alpha$函数决定$\beta$，记为$\alpha\to\beta$，读作"$\alpha$函数依赖于$\beta$"在$R$上成立
		- 非形式化的，函数依赖指属性集$\alpha$的值可以决定属性集$\beta$的值
		    - 例：![[Pasted image 20260611160739.png]]在这个关系模式中，$A\rightarrow B$不成立；但$B\rightarrow A$成立，因为$B$的值可以唯一确定$A$的值
	- 几何意义：
		- 实例（Instance）：某个关系模式的一个具体表格，包含了若干元组（Tuple）
		- 实例空间（Instance Space）：所有可能的实例组成的集合
			- 例：![[Pasted image 20260611161832.png]]
		- 函数依赖约束子集：满足函数依赖$\alpha\to\beta$的实例空间中的实例集合
			- 例：![[Pasted image 20260611161905.png]]
	- 应用：
	    - 超键的函数依赖定义：$K$是$R$的超键，当且仅当$K\rightarrow R$成立，即$K$的值可以唯一确定关系模式$R$中的所有属性的值
	    - 候选键的函数依赖定义：$K$是$R$的候选键，当且仅当$K\rightarrow R$成立，并且对于$K$的任何真子集$K'$，$K'\not\rightarrow R$，即候选键是最小的超键
	    - 检验关系的合法性：函数依赖定义了对应实例空间中的合法子集，只有满足函数依赖约束的实例才是合法的关系实例
		    - **满足**（Satisfied）：针对**单个实例**的描述，如某个具体的表格满足函数依赖约束
			    - 例：![[Pasted image 20260611162138.png]]上述关系实例满足`loan-number → customer-name`
			- **保持**（Hold on）：针对**整个关系模式**的描述，当$R$的所有合法实例都满足依赖$\alpha\to\beta$时，称$\alpha\to\beta$在关系模式$R$上保持
	    - 函数依赖与一致性约束：函数依赖可以刻画部分属性决定另一部分属性的业务规则
		    - 例：贷款信息模式`Loan-info-schema = (customer-name, loan-number, branch-name, amount)`
			    - 成立的函数依赖：
	                - `loan-number → amount`：贷款号可以唯一确定贷款金额
	                - `loan-number → branch-name`：贷款号可以唯一确定贷款所在的分行
	            - 不成立的函数依赖：
	                - `loan-number → customer-name`：贷款号无法唯一确定客户姓名，因为一个客户可能有多个贷款
	- 平凡函数依赖（trivial functional dependency）：如果$\beta\subseteq\alpha$，则$\alpha\to\beta$是平凡函数依赖
		- 这一类函数依赖没有实际意义，因为$\alpha$的值自然包含了$\beta$的值，没有约束作用
		- 后续所有讨论的函数依赖默认是非平凡函数依赖
- 函数依赖的闭包：
	- 定义：给定一组原始函数依赖$F$，所有可以通过$F$推导出来的函数依赖的集合称为$F$的闭包，记为$F^+$，即$F^+=\{\alpha\to\beta|\alpha\to\beta\text{由}F\text{推导而来}\}$
		- 设计数据库时，实际的约束是函数依赖的闭包，而不仅仅是原始定义的函数依赖
	- Armstrong公理：推导函数依赖的基本规则：
        1. 自反律（Reflexivity）：如果$\beta\subseteq\alpha$，则$\alpha\to\beta$成立
        2. 增广律（Augmentation）：如果$\alpha\to\beta$成立，则$\gamma\alpha\to\gamma\beta$成立
        3. 传递律（Transitivity）：如果$\alpha\to\beta$成立，且$\beta\to\gamma$成立，则$\alpha\to\gamma$成立
    - 衍生规则（由公理推导而来）：
	    1. 合并规则（Union）：如果$\alpha\to\beta$成立，且$\alpha\to\gamma$成立，则$\alpha\to\beta\gamma$成立
	    2. 分解规则（Decomposition）：如果$\alpha\to\beta\gamma$成立，则$\alpha\to\beta$成立，且$\alpha\to\gamma$成立
	    3. 伪传递规则（Pseudotransitivity）：如果$\alpha\to\beta$成立，且$\gamma\beta\to\delta$成立，则$\alpha\gamma\to\delta$成立
	- 闭包的计算：以关系模式：$R=(A,B,C,G,H,I)、F=\{A\to B,A\to C,CG\to H,CG\to I,B\to H\}$为例，计算$F$的闭包$F^+$：
		- 传递律$A\to B$，$B\to H$，得到$A\to H$
		- 增广律$CG\to I$，对两边加$A$，得到$ACG\to AI$
		- 用合并律$CG\to H$和$CG\to I$，得到$CG\to HI$

- 函数依赖的冗余与标准覆盖
	- 冗余的函数依赖：可以被其他函数依赖推导出来的函数依赖
		- 例：$F={A→B,B→C,A→C}$中，$A→C$是冗余的，因为它可以通过$A→B$和$B→C$推导出来
	- 函数依赖的冗余属性：
		- 左边冗余：如$F={A→C,AB→C}$中，$AB→C$中的$B$是冗余的，因为$A→C$已经成立
		- 右边荣誉：如$F={A→C,AB→CD}$中，$AB→CD$中的$C$是冗余的，因为$A→C$已经成立
		- 冗余属性的测试：
			- 左边冗余：要判断$A\in \alpha$是否是$\alpha\to \beta$中的冗余属性，计算$(\alpha-\{A\})^+$，如果结果包含$\beta$，则$A$是冗余的
			- 右边冗余：要判断$A\in\beta$是否是$\alpha\to \beta$中的冗余属性，构造$F'=(F-\{\alpha\to \beta\})\cup\{\alpha\to (\beta-\{A\})\}$，计算$\alpha^+$，如果结果包含$A$，则$A$是冗余的
	- 标准覆盖（canonical cover）：标准覆盖$F_c$是和$F$等价的最小函数依赖集合，满足如下条件：
		- $F\Leftrightarrow F_c$，即$F$和$F_c$的闭包相同，表示它们定义了同样的约束
		- $F_c$中没有冗余的函数依赖
		- $F_c$中每个函数依赖都没有冗余属性
		- $F_c$中左边相同的函数依赖被合并
		- 计算：![[Pasted image 20260611165437.png]]
			- 合并左侧相同的函数依赖：如$A→B$和$A→C$合并成$A→BC$
			- 消除左边冗余属性：如$AB→C$中$A$是冗余的，消除后得到$B→C$
			- 消除右边冗余属性：如$A→BC$中$B$是冗余的，消除后得到$A→C$
# 模式分解
- 关系分解（Decomposition）：将一个关系模式$R$分解成多个关系模式$\{R_1,R_2,\cdots,R_n\}$的过程，分解必须满足以下条件：
	- 每个关系$R_i$都是
	- 分解必须是**无损连接**的
	- 分解尽可能**保持依赖**
	- 基本原则：分解后的所有关系的属性的并集必须等于原关系的属性集合，即$\bigcup_{i=1}^n R_i = R$
- 无损连接分解（Lossless Join Decomposition）：
	- 无损连接：对关系模式$R$的所有可能实例$r$，满足$r=\Pi_{R_1}(r)\bowtie \Pi_{R_2}(r)\bowtie \cdots \bowtie \Pi_{R_n}(r)$，其中$\Pi_{R_i}(r)$表示关系实例$r$在关系模式$R_i$上的投影，$\bowtie$表示自然连接
		- 非形式化地，无损连接分解表示把原关系投影到分解后的模式上，再自然连接回来，必须能完全还原原关系，不能有数据的改变
		- 例：一个有损连接分解![[Pasted image 20260611171615.png]]
		- 二元分解判定定理：关系$R$分解为$R_1$和$R_2$是无损连接的，当且仅当$R_1\cap R_2\to R_1$或$R_1\cap R_2\to R_2$在$F^+$中成立
			- **两个分解后的关系，其公共属性集必须是其中某个关系的超键**
	- 无损连接分解是必须满足的条件，否则数据将会出现丢失或错误
- 依赖保持分解（Dependency Preserving Decomposition）：
    - 依赖保持：设$F_i$是$F^+$中所有属性都包含在$R_i$中的函数依赖（也就是$F$在$R_i$上的投影），如果$\bigcup_{i=1}^n F_i$的闭包等于$F^+$，则称分解$\{R_1,R_2,\cdots,R_n\}$保持依赖
        - 非形式化地，依赖保持分解表示原模式中的所有约束，都能在分解后的单个关系中检查，不需要将多个关系连接起来验证
    - 意义：若不保持依赖，则每次修改数据时，为验证依赖，必须做连接操作，降低效率
	- 依赖保持分解是可选的条件，但在实际设计中，通常希望满足依赖保持，以提高数据操作的效率
