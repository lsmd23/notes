- 为保证事务的原子性和持久性，必须针对系统可能发生的故障，进行合理的恢复
# 故障分类
- 事务故障（Transaction Failure）：事务执行过程中发生的错误
	- 逻辑错误（Logical Errors）：事务因为自身的内部错误无法正常完成，如输入数据无效，计算溢出，超出资源限制等
	- 系统错误（System Errors）：数据库系统必须主动终止正在执行的事务的错误，如死锁回滚等因并发调度引起的事务中止
- 系统崩溃（System Crash）：数据库系统发生崩溃，导致系统突然停止运行
	- fail-stop假设：系统崩溃只会导致内存中的数据丢失，而不会损失非易失性存储介质（如磁盘）上的数据
	- 数据库系统会通过完整性校验防止磁盘数据被意外破坏
- 磁盘故障（Disk Failure）：磁盘发生故障，导致存储在磁盘上的数据丢失或损坏
	- 对数据有不可逆的破坏，无法通过数据库系统的恢复机制来修复
	- 数据库系统会通过冗余存储和备份机制来防止数据丢失
- 故障的问题：
	- 原子性问题：事务的修改是分布进行的，若故障发生在事务执行的中途，则可能导致事务的修改只完成了一部分，造成数据的不一致
	- 持久性问题：事务的修改可能已经完成，但由于系统崩溃或磁盘故障，导致修改的数据无法持久化到磁盘上，造成数据丢失
- 恢复的思路：
	- 事务正常执行时，提前收集记录恢复用的信息，为故障后的恢复做准备（也即基于日志的恢复算法）
	- 故障发生后需要进行恢复操作
# 数据的存储与传输
- [[L3 存储与文件结构|数据的存储]]
- 事务访问数据的流程：
	- 示意图：![[Pasted image 20260529192703.png]]
	- 核心步骤1：磁盘与缓存间的数据传输
		- 物理块：存放在磁盘上的数据块，处于持久化存储状态
		- 缓存块：临时存放在内存里的数据块，处于易失性存储状态
		- `input(B)`：将磁盘上的物理块B读取到内存中，形成缓存块B
		- `output(B)`：将内存中的缓存块B写回磁盘，形成物理块B
		- *注：对磁盘的I/O往往是性能瓶颈，因此数据库系统会尽量减少磁盘I/O的次数*
	- 核心步骤2：缓存与事务工作区之间的数据传输
		- 工作区：事务执行过程中用于存储临时遍历的区域，事务的读写操作本质都是在私有的工作区中进行的
		- `read(B)`：将缓存块B中的数据`X`读取到事务的工作区`xi`中
			- 事务第一次访问数据项时，必须进行这一步操作
		- `write(B)`：将事务工作区`xi`中的数据`X`写回缓存块B中
			- 这一步并不会修改磁盘上的数据，直到缓存块被写回磁盘时才会修改物理块
# 日志与恢复机制
- 基于日志的恢复：
	- 目标：保证事务的原子性，事务不能出现部分修改成功的情况
	- 日志（log）：数据库系统为了记录事务执行过程中修改的数据项的原始值和修改后的新值而生成的一系列记录
	- 为保证原子性，必须先在稳定的存储介质上记录日志，再修改数据库本身
	- **WAL原则**：为保证故障时的恢复，数据库系统在事务执行过程中会先将修改的信息写入稳定的存储介质中，再修改数据库本身（Write-Ahead Logging）
	- 日志的生成：
		- 事务开始时，写一条`<T_i start>`日志记录，标记事务的开始
		- 事务修改数据前，写一条记录`<T_i, X, V_1, V_2>`，其中：
			- `X`：被修改的数据项
			- `V_1`：修改前的值
			- `V_2`：修改后的值
		- 事务提交时，写一条`<T_i commit>`日志记录
			- **只有在提交日志被写入磁盘后，事务才被认为是提交成功的**，此时事务的修改才具有持久性
- 数据库修改的策略：
	- 延迟修改：
		- 规则：事务对数据库的修改只有在提交时才写入磁盘
		- 优点：恢复逻辑简单，只需丢弃即可
		- 缺点：日志中存储所有的修改副本，一次性向磁盘写入可能有I/O压力
	- 立即修改：
		- 规则：事务对数据库的修改在执行过程中就写入磁盘
		- 更新日志必须在修改数据库之前写入磁盘，以保证原子性
		- 刷入磁盘的时间相对更加灵活，降低了磁盘的I/O压力
			- 例：![[Pasted image 20260608153357.png]]$B_C$在$T_1$提交前就写入磁盘，$B_A$在提交后才写入磁盘
	- 并发事务的约束：
		- 多事务并发时，共享同一磁盘缓冲区和日志文件，可能会导致日志顺序的混乱
		- 调度规则：为保证并发事务的正确性所必须的规则约束
			- **如果事务$T_i$修改了数据项$X$，那么在$T_i$提交或中止前，其他事务不能修改$X$**
			- 实现：严格两阶段锁协议，也即事务在修改数据项时加排他锁直到事务结束
- 恢复的基本操作：
	- Undo：事务的撤销
		- 作用：回滚事务所有的修改，恢复到事务执行前的状态，**自后向前执行**（going backwards）
		- 操作：
			- 对日志记录`<T_i, X, V_1, V_2>`的Undo：将旧值`V_1`写回数据项`X`中
			- 额外日志：每次恢复数据项`X`到旧值`V`时，需写补偿记录`<T_i, X, V>`，在完成后记录`<T_i, abort>`，标记事务的中止
	- Redo：事务的重做
		- 作用：将事务修改重新应用，确保提交的修改全部执行，**自前向后执行**（going forwards）
		- 操作：
			- 对日志记录`<T_i, X, V_1, V_2>`的Redo：将新值`V_2`写回数据项`X`中
			- Redo操作不写新的日志记录，仅重新应用修改
	- 判断逻辑：
		- 需要Undo的事务：日志中有`<T_i start>`但没有`<T_i commit>`或`<T_i abort>`的事务，说明事务启动后未正常结束，需要执行Undo操作
		- 需要Redo的事务：日志中有`<T_i commit>`的事务，说明事务已提交，需要执行Redo操作以确保修改被应用
	- 撤销事务的“重复历史”：
		- 事务$T_i$若已经执行过Undo，写入`<T_i abort>`ilu，之后发生故障，则恢复时会被再次Redo
		- 这一步骤产生了冗余操作，但必要的重复历史是日志恢复的必备记录

- 性能优化：检查点（Checkpoint）机制
	- 如果每次故障都从头开始扫描，日志文件将无限制增长，导致恢复复杂度过高
	- 检查点：DBMS系统周期性主动执行的一套操作
		- 将内存日志缓冲区中，所有未落盘的日志，全部写入稳定存储（WAL规则）
		- 将内存日志缓冲区中所有被修改过的脏数据块，强制刷入磁盘
		- 在日志中写入一条特殊的检查点记录`<checkpoint L>`，其中`L`是检查点执行时系统中所有活跃事务的列表
			- 活跃事务指已经开始但还没有提交/中止的事务
	- 带检查点的恢复规则：
		1. 定位最近的检查点：从后向前扫描日志，找到最近的检查点记录
		2. 对事务分类：
			- 可忽略的事务：在检查点之前就已经完成的事务
			- 需处理的事务：检查点列表里的活跃事务以及检查点之后才创建的事务
	- 日志的清理：
		- 由于检查点机制的存在，日志文件中可以删除检查点之前的一些记录压缩日志空间
		- 规则：
			1. 沿日志向前扫描，为检查点列表`L`中的每一个活跃事务找到对应的事务启动日志`T_i start>`
			2. 找到这些启动日志中最靠前的一条
			3. 删除这条日志之前的所有日志记录，因为这些记录对应的事务都已经完成，不再需要用于恢复
	- 例：![[Pasted image 20260608173620.png]]
- 标准恢复算法：
	- 事务故障：单个事务异常导致的回滚
		1. 从后向前扫描日志
		2. 遇到修改记录，执行Undo（恢复旧值，写入补偿日志）
		3. 扫描到对应事务的启动日志`<T_i start>`为止
		4. 写入`<T_i abort>`
	- 系统崩溃：内存全部丢失的场景，分两阶段执行
		- 阶段一：Redo重做阶段（补全已结束的事务）
			1. 找到最近的检查点`<checkpoint L>`，初始化`undo-list`为`L`
			2. 从检查点开始向后扫描日志
				- 修改日志，执行Undo
				- `<T_i commit>/<T_i abort>`，将`T_i`从`undo-list`中移除
				- `<T_i start>`，将`T_i`加入`undo-list`
		- 阶段二：Undo回滚阶段（撤销未结束的事务）
			1. 从后向前扫描日志
			2. 若日志记录对应的事务在`undo-list`中，执行Undo
			3. 扫描到事务开始日志时，写入`<T_i abort>`，将`T_i`从`undo-list`中移除
			4. 当`undo-list`为空时，结束
- 日志记录缓存：
	- 若日志每次都刷入磁盘，I/O瓶颈会导致性能降低，因此实际工程实现会使用缓冲机制
		- 当缓冲区写满时，进行日志刷盘
		- 为确保事务的日志可以正常记录，`log force`操作可以强制进行日志的刷盘
	- 缓存机制下的先写日志规则（WAL）：
	    - 日志必须按照创建顺序刷入存储介质
	    - 事务只有在提交日志`<T_i commit>`被写入磁盘后才被认为是提交成功的
	    - 日志的刷盘应当先于数据的刷盘
	- 缓存机制：参见[[L3 存储与文件结构#^21f442|存储与文件结构——缓存]]
	- 缓存的策略：
		- 非强制策略与强制策略：事务提交时，是否强制将日志刷入磁盘
			- 强制策略（Force）：事务提交时强制将日志刷入磁盘
				- 恢复时很少Redo
				- 提交时集中I/O，性能较差
			- 非强制策略（No-force）：事务提交时不强制刷盘，允许日志在内存中等待
				- 恢复时需要Redo更多的日志记录
				- 提交时性能较好，但可能导致更长的恢复时间
		- 窃取策略与非窃取策略：事务是否允许未提交事务的脏块刷盘
			- 非窃取策略（No-steal）：事务不允许未提交事务的脏块刷盘
				- 磁盘上不会有未提交的修改，恢复时很少Undo
				- 脏块不必要占用缓存
			- 窃取策略（Steal）：事务允许未提交事务的脏块刷盘
				- 缓存可以淘汰脏块，利用率高
				- 但磁盘上会有未提交的修改，需要Undo回滚
		- 目前被广泛使用的是**非强制策略和窃取策略**的组合，这样可以兼顾性能和恢复效率
			- 代价是恢复时必须支持Undo和Redo两种操作
	- 缓存的刷盘：必须确保缓存刷盘保持数据的一致性
		1. 加排他闩（latch）：对缓存块加排他闩，防止其他事务修改缓存块（闩在刷盘结束后就会释放）
		2. 日志刷盘：将缓存块对应的日志记录刷入磁盘，确保日志先于数据刷盘（WAL原则）
		3. 数据刷盘：将缓存块写回磁盘，形成物理块
		4. 释放排他闩
	- 缓存的实现：
		- 预留物理内存：操作系统提前给数据库分配专用的物理内存作为缓存
			- 内存固定，不够灵活，无法动态调整
		- 虚拟内存：数据库缓存置于操作系统的虚拟内存内，由OS进行动态管理
			- 内存利用率高
			- 存在**双页问题**（Dual Paging Problem）
		- 双页问题：
			- 虚拟内存的工作机制是，当内存不够用时，OS会将修改过的脏页写入磁盘的交换空间内
			- 数据库缓存的问题：
				- 操作系统会将数据库的脏缓存页写入swap空间
				- 数据路需要把脏页刷入数据库磁盘时，需要从swap空间读取脏页再写入，额外多了两次I/O、
			- 解决：使用预留的物理内存，或自行实现缓存淘汰，避免调用OS的虚拟缓存机制
# 磁盘故障与远程备份
- 磁盘故障：磁盘发生故障，导致存储在磁盘上的数据丢失或损坏
    - 由于日志基于磁盘的持久化特性，磁盘故障会导致日志丢失，无法进行基于日志的恢复
- 数据转储（Dump）：
	- 定义：也叫全量备份，定期将整个数据库的内容拷贝到安全的稳定存储（如备份磁盘，云存储等）上
	- 步骤：
        1. 暂停新事务，确保没有活跃事务
        2. 刷空日志缓冲区，确保所有日志记录都已写入磁盘
        3. 刷空脏数据，确保没有未持久化的数据在缓冲区中
        4. 拷贝数据库文件和日志文件
        5. 写入`<dump>`标记，标记这次备份
    - 恢复流程：
	    1. 还原最近一次的全量备份：将备份的数据库文件恢复到新的磁盘上
	    2. 重做备份之后的所有事物：从`<dump>`标记之后的日志开始扫描，Redo所有提交的事务
	- 转储的类型：
		- 物理转储：拷贝所有的物理文件本身
			- 速度快，但依赖平台
		- 导出SQL语句：将数据库中的数据导出为SQL语句的形式，恢复时执行这些SQL语句来重建数据库
			- 平台无关，但速度较慢
- 远程备份系统：
    - 定义：将数据库的备份存储在远程位置，如云存储或远程服务器上，以防止本地磁盘故障导致的数据丢失
	    - 示意图：![[Pasted image 20260608195549.png]]
		    - 主站点：正常服务的数据库，处理业务请求
		    - 备站点：部署在异地的备份数据库，不直接处理业务，只同步数据
	- 热备份（Hot-spare）：
		- 正常运行时，备站点实时同步日志；备站点接收Redo日志后，立即应用日志进行数据更新
		- 主站点故障时，备站点快速接管，回滚未完成的事务，并切换为新的主站点对外进行服务
