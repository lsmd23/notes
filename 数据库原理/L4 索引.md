- 数据库需要将SQL语句翻译成关系代数，之后去磁盘上扫描对应的数据文件，再返回给用户
- 磁盘的I/O是数据库的瓶颈，因此良好的索引设计对数据库性能至关重要
# 基本概念
- 查找键（Search Key）：用于在文件中查找记录的属性（或属性集合）
	- 例：查找`ID = 21122`的学生，则`ID`就是查找键
- 索引文件（Index File）：由**索引项（Index Entries）**组成的文件，每个索引项的结构为`[search-key | pointer]`
	- `search-key`：索引项中存储的查找键值
    - `pointer`：指向数据文件中对应记录磁盘地址的指针
- 索引的基本类型：
	- 顺序索引（Ordered Indices）：
		- 查找键按照**排序顺序**存储在索引文件中
		- 代表：[[L4 索引#B+树索引|B+树索引]]
		- 支持等值查询，也支持高效的范围查询
	- 哈希索引（Hash Indices）：
        - 根据查找键的值通过哈希函数计算存储位置，索引项分布在不同的桶中
        - 等值查询极快，但不支持范围查询，同时需要处理哈希冲突
- 索引的评价指标：
    - 支持的访问类型：是否支持等值查询、范围查询等
    - 时间开销：索引查找的访问时间，插入时间，删除时间等
    - 空间开销：索引占用的存储空间
# 顺序索引
- 分类：
	- [[L4 索引#^408096|主索引]]（Primary Index）：查找键决定数据文件物理顺序的索引，也叫聚集索引（Clustering Index）
		- 数据文件按照主索引查找键物理排序
		- 查找键通常是主键，但不是必须
		- 也叫一级索引
	- [[L4 索引#^adaafc|次级索引]]（Secondary Index）：查找键不决定数据文件物理顺序的索引，也叫非聚集索引（Non-clustering Index）
        - 数据文件按照其他属性排序，次级索引存储该属性的值和对应记录的地址
        - 用于非主键字段的查询
        - 也叫二级索引
- 主索引： ^408096
	- 稠密索引（Dense Index）：
        - 每个数据记录对应一个索引项，索引项存储查找键值和对应记录的地址
	        - 例：按`ID`的索引![[Pasted image 20260418151029.png]]按`dept_name`的索引![[Pasted image 20260418151055.png]]
	    - **稠密索引项的数量等于索引键值的数量**，若出现“一对多”的情况，索引只存储一个索引项，指向第一个对应记录的地址，后续记录通过指针链访问
	    - 查询效率高，但占用空间大
	    - 删除：删除数据不一定引起索引项的删除，只有当被删除记录是索引项指向的最后一个记录时，才需要更新索引项指向下一个对应记录的地址
        	- 例：![[Pasted image 20260418152744.png]]删除`Comp.Sci.`对应的一个记录后，索引项仍然存在，只有当删除`Comp.Sci.`对应的最后一个记录时，才需要更新索引项指向下一个对应记录的地址
        - 插入：先查找是否有对应的查找键值的索引项，如果有，插入记录后更新索引项指向新记录的地址；如果没有，插入记录后创建新的索引项
        	- 例：![[Pasted image 20260418153027.png]]
	- 稀疏索引（Sparse Index）：
        - 只为部分数据记录建立索引项，通常是每块数据记录的第一个记录（也即**数据块**的第一个记录）
	        - 例：![[Pasted image 20260418151327.png]]查询时，从`32343`位置开始，直到查找到`45565`的位置
	    - 稀疏索引项的数量小于索引键值的数量，一般和数据块数量保持一直（减少I/O），占用空间小，但查询效率较低，尤其是当数据块内记录较多时
	    - 删除：若删除的是第一个记录，需要更新索引项指向下一个记录的地址；若删除的不是第一个记录，不需要更新索引项
		    - 例：![[Pasted image 20260418152918.png]]删除`32343`对应的记录后，需要更新索引项指向下一个记录的地址；删除`45565`对应的记录后，不需要更新索引项
		- 插入：若插入后没有产生新的数据块，不需要更新索引项；若插入后产生了新的数据块，需要创建新的索引项指向新数据块的第一个记录
- 次级索引：非主键字段查询 ^adaafc
    - 由于次级索引不决定数据文件的物理顺序，因此**次级索引必须是稠密索引**，每个数据记录都需要一个索引项来存储对应的地址
    - 例：利用指针桶存储每个次级索引对应的记录地址![[Pasted image 20260418151831.png]]
- 多级索引：当数据文件较大时，单级索引的查询效率会下降，因此可以使用多级索引来提高查询效率
    - 将主索引当作顺序文件，再建立一层稀疏索引
	    - 将这一层稀疏索引称为**外层索引（Outer Index）**
	    - 原来的主索引称为**内层索引（Inner Index）**
    - 例：![[Pasted image 20260418152433.png]]查询时，从外层索引开始。由于外层索引可以直接装入内存，因此访问外层索引的I/O开销较小，之后根据外层索引的指针访问内层索引，再根据内层索引的指针访问数据文件，整体查询效率得到提升
- 索引的代价：
	- 索引的更新开销：插入、删除、修改数据记录时，需要同时更新索引项，尤其是当索引项较多时，更新开销较大
	- 次级索引的扫描效率：由于次级索引与数据文件的物理顺序不一致，扫描次级索引时可能需要频繁访问数据文件，导致I/O开销较大
# B+树索引
- 当数据文件动态增长时，产生大量的溢出数据块，需要定期重构整个文件才能恢复性能。B+树可以自动局部调整数据结构，使其保持高性能，避免频繁重构文件
	- 参见[[L8 高级搜索树#B树|数据结构——B树]]
- 基本性质：
	- 根节点到所有叶子节点的路径长度相同，也即**所有叶子节点深度相同**，保证所有查询的磁盘I/O次数稳定，时间复杂度为$O(\log n)$
